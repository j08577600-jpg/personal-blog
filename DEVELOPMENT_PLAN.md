# 开发计划 v2

> 日期：2026-03-25
> 状态：V2 规划版
> 说明：`docs/DEVELOPMENT_PLAN_v1.md` 归档保存 V1 计划；本文件作为 V2 的当前执行入口。

## 一、V2 总目标

V2 不再以“补齐基础能力”为主，而是聚焦三件事：

- 写得更快：降低写作、编辑、发布的操作成本
- 找得更准：增强搜索、筛选、内容发现能力
- 规模变大后不乱：统一内容模型与管理方式，避免内容类型越加越散

---

## 二、阶段划分

### 阶段 A：写作效率提升（最高优先级）

目标：把当前文件驱动内容系统进一步打磨成高频可用的写作工作台。

#### 已有基础
- 在线写作工作台 V2 已上线
- 文章编辑 / 新建 / 发布流程已打通
- 草稿 / 已发布状态管理已完成

#### 本阶段待做
- 自动保存与更明确的保存反馈
- 编辑恢复能力（异常退出 / 刷新后的恢复策略）
- 发布确认与防误操作体验优化
- frontmatter 编辑体验继续打磨（slug / excerpt / tags / cover）
- 写作页信息密度、交互顺序、错误提示继续优化
- 文章版本差异 / 最近一次备份的可读化展示（轻量方案）

#### 验收标准
- 写文章时不需要频繁担心丢内容
- 新建、编辑、发布的路径更顺手
- 常见错误（字段缺失、格式错误、重复 slug）提示更直接

---

### 阶段 B：认证扩展与作者入口完善（高优先级）

目标：在不引入复杂账户系统的前提下，扩展登录入口，提升作者使用便利性。

#### 本阶段待做
- 接入 Google OAuth
- 接入 Microsoft OAuth
- 统一登录入口文案与交互
- 保持作者权限仍按 email 白名单判断
- 验证同邮箱不同 provider 的行为一致性

#### 明确边界
- 不做账号中心
- 不做复杂账号合并 UI
- 不引入数据库型用户系统
- 不改变现有作者白名单主逻辑

#### 验收标准
- Google 可正常登录
- Microsoft 可正常登录
- 白名单邮箱登录后仍可进入作者后台
- 非白名单邮箱即使 OAuth 成功也不能进入作者功能

---

### 阶段 C：内容模型统一（高优先级）

目标：把 `posts / projects / reading-list / notes` 逐步整理成一套一致、低心智负担的内容结构。

#### 本阶段待做
- 梳理各内容类型的公共字段与差异字段
- 统一索引字段、列表展示字段、搜索字段
- 统一各类型的状态与可见性语义
- 明确哪些类型进入站内搜索，哪些先保留独立展示
- 减少“同站点不同内容类型各写一套规则”的维护成本

#### 明确边界
- 继续以文件驱动为主
- 不为了未来扩展提前抽象数据库层
- 不在 V2 前期直接引入完整 CMS

#### 验收标准
- 新增内容类型时，不需要再从头设计一遍展示与索引逻辑
- 搜索、列表、详情页的数据结构更一致
- 内容类型扩展不会显著增加维护成本

---

### 阶段 D：搜索增强（中高优先级）

目标：让搜索从“能用”升级到“更适合真实内容站点使用”。

#### 已有基础
- 搜索已上线
- URL 查询参数持久化已完成
- `posts + projects` 已纳入当前搜索主路径

#### 本阶段待做
- 评估并接入 `notes / reading-list`
- 增加按类型 / 标签等维度的筛选
- 优化中文短 query 的召回与排序
- 清理现存低优先级代码异味
- 继续优化搜索结果展示的一致性

#### 验收标准
- 搜索结果更稳定
- 内容类型增多后仍然可找
- 搜索页能承担站内“内容入口”的角色，而不只是演示功能

---

### 阶段 E：发布运营能力（中优先级）

目标：让站点从“可发布”走向“可持续运营”。

#### 本阶段待做
- 发布前 checklist
- 延迟发布 / 定时发布（如确认有真实需求）
- 首页推荐位 / 置顶机制
- 系列文章 / 专题组织能力
- 更新记录与内容维护动作支持

#### 验收标准
- 发文流程更可控
- 首页内容组织更主动
- 内容增长后能体现运营策略，而不是简单堆列表

---

### 阶段 F：基础数据与反馈（中优先级）

目标：建立最小可用的数据反馈，避免内容运营完全靠感觉。

#### 本阶段待做
- 明确最小分析指标（页面访问、来源、热门内容、搜索关键词等）
- 选择轻量实现方案
- 避免为分析而引入过重基础设施

#### 验收标准
- 能知道哪些内容被访问、被点击、被搜索
- 能支撑内容选题与首页编排的判断

---

## 三、优先级顺序

### 第一批（先做）
- 阶段 A：写作效率提升
- 阶段 B：认证扩展与作者入口完善

### 第二批（紧接着做）
- 阶段 C：内容模型统一
- 阶段 D：搜索增强

### 第三批（在前两批稳定后）
- 阶段 E：发布运营能力
- 阶段 F：基础数据与反馈

---

## 四、当前已明确纳入计划的任务

- OAuth 扩展：接入 Google 登录
- OAuth 扩展：接入 Microsoft 登录
- 登录入口与白名单口径整理
- 写作工作台体验继续打磨
- 多内容类型统一建模
- 搜索接入更多内容类型并增强筛选能力

---

## 五、暂不在 V2 前期做的事

- 数据库驱动内容系统
- 完整 CMS
- 复杂用户系统 / 账号中心
- 为未来扩展过度抽象内容层
- 为搜索一次性引入过重方案

---

## 六、执行原则

- 继续遵守 `docs/OpenClaw_项目协作规范_v1.md`
- 复杂功能先方案设计，再执行计划，再进入开发
- 优先做高频使用、强体感、低歧义的能力
- 每新增一类复杂度，都要先判断是否真的带来使用价值
- V2 的核心不是“功能更多”，而是“系统更顺手、更稳定、更能长期写下去”
