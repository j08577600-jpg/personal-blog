# OpenClaw 开发工作流模板（可复用版）

> 用途：把这份文档交给其他 agent，让它们在别的项目中生成一套**类似的 OpenClaw 项目开发工作流**。
> 
> 目标：统一角色分工、阶段流转、文档产物、并行策略与发布收口方式，避免出现“无计划开发”“builder 代写计划”“阶段汇报代替流程推进”“一个 agent 包办全流程”等常见失控情况。

---

## 一、工作流目标

这套工作流适用于：
- 中大型功能开发
- 多 agent 协作
- 需要方案设计、执行计划、代码实现、交叉审查、测试验收、发布收口的项目
- 需要在 OpenClaw 中长期稳定复用的开发流程

核心原则：
- **先 Plan，后开发**
- **先过审，再测试**
- **先测试通过，再发布**
- **角色边界清晰**
- **功能可以并行，但功能内部流程不能乱**

---

## 二、角色分工（推荐标准版）

### 1. planner
负责：
- P1 方案设计
- P2 执行计划
- P6 最终收口
- 架构一致性判断
- 最终报告整理

### 2. designer
负责：
- UI / 交互方向提案
- 页面结构
- 视觉方向
- 文案方向（若项目包含品牌语气或页面文案重设计）

### 3. builder
负责：
- P3 实现开发
- 按 P2 执行
- 在执行过程中更新计划里的进度、决策日志、验证记录

### 4. reviewer
建议至少 2~3 审：
- 主审：技术正确性、实现质量、与方案一致性
- 二审：体验、盲点、边界情况、文案一致性
- 三审：风险、结构、维护性、逻辑严密性

### 5. tester
负责：
- P5 测试验收
- 输出测试报告
- 判定是否允许发布

---

## 三、硬性角色规则

### 1. `planner` 才能写 P1 / P2
- `方案设计.md` 的作者必须是 `planner`
- `执行计划.md` 的初稿作者必须是 `planner`

### 2. `builder` 不得代写 P2
- builder 只能执行 P3
- builder 可以更新执行计划中的：
  - 进度
  - 决策日志
  - 验证记录
- builder **不得擅自改写**：
  - 目标
  - 范围
  - 验收标准
- 若计划失效或范围变化，必须退回 `planner`

### 3. `tester` 只负责测试验收
- tester 不负责代替审查
- tester 不负责代替 planner 收口

### 4. `planner` 负责最终收口
- 功能通过测试后，必须回到 `planner`
- planner 负责判断：
  - 文档是否齐全
  - 交付是否闭环
  - 是否进入 P6 最终报告

---

## 四、阶段流转（标准主流程）

### P1 方案设计
输出：`方案设计.md`

内容至少包含：
- 目标
- 方案结论
- 边界与约束
- 风险与取舍
- 不做什么

### P2 执行计划
输出：`执行计划.md`

内容至少包含：
- 上下文
- 实施步骤
- 验收标准
- 风险点
- 恢复方式
- 决策日志位置

### P3 实现开发
输出：代码 + 执行记录

要求：
- 必须按 P2 执行
- 每个停止点更新：
  - 进度
  - 决策日志
  - 验证记录

### P4 交叉审查
输出：`审查报告.md`

要求：
- 主审 / 二审 / 三审可并行
- 审查必须明确区分：
  - blocking issues
  - non-blocking issues
- 未过审不得进入测试

### P5 测试验收
输出：`测试报告.md`

要求：
- 至少覆盖：
  - 单元测试
  - 冒烟测试
  - 功能测试
  - 回归测试
- 必须明确：
  - 通过项
  - 失败项
  - 是否允许发布

### P6 发布 / 合并
输出：
- `P6_最终报告.md`
- `P6_最终报告.pdf`（推荐作为最终交付物）

要求：
- 收集 P1/P2/P4/P5 文档
- 汇总最终报告
- Git 合并
- 发布部署
- 更新阶段计划/开发计划文档

## 五、阶段自动推进规则

这是最容易被 agent 执行歪的地方，必须写死。

### 核心规则
**一旦某个功能进入阶段流转，在用户未叫停的前提下，编排者必须自动推进到下一角色。**

也就是：
- planner 完成 P1/P2 → 自动进 builder
- builder 完成 P3 → 自动进 reviewer
- reviewer 通过 P4 → 自动进 tester
- tester 通过 P5 → 自动回 planner 收口
- planner 收口完成 → 自动补齐 P6 需要的最终报告 / PDF / 合并 / 发布动作

### 禁止行为
- 不能把“进度汇报”当成“暂停点”
- 不能每完成一个阶段就停下来等用户再次催
- 不能把“planner 收口完成”误说成“P6 全完成”

### 汇报口径必须区分清楚
必须明确区分这几种状态：
- `P1/P2 完成`
- `P3 完成`
- `P4 完成`
- `P5 通过`
- `planner 收口完成`
- `P6_最终报告.md 已生成`
- `P6_最终报告.pdf 已生成`
- `已合并到 main`
- `已重新部署`

不能混说。

---

## 六、多功能并行策略

### 并行单位
**并行单位 = 不同功能**

不是：
- 同一个功能的不同阶段并行
- 一个功能里一个 agent 包办所有角色

### 标准规则
- 当多个独立功能同时开发时，最多开启 **4 路子 agent**
- 超过 4 个功能时，其余功能排队
- 每个功能内部仍按角色流转：
  - `planner -> builder -> reviewer -> tester -> planner`

### 两条简化约束
- 明显属于同一个功能的，不硬拆
- 明显会改同一个核心文件的，不同时开做

### 关键禁令
不允许：
- 一个子 agent 包办同一功能的完整流程
- 把“功能并行”偷换成“单 agent 一路跑到底”

---

## 七、UI 类任务的特殊流转

当任务涉及 UI / UX / 文案方向时，建议使用：

`designer -> planner -> builder -> reviewer -> tester -> planner`

### 原因
- 设计方向先发散
- 再由 planner 收敛为正式方案和执行计划
- 再进入实现

### 文案处理规则
如果文案是 UI 重设计的一部分：
- 文案不是附属项
- 要纳入 P1 / P2
- 要成为验收标准的一部分
- 允许在计划中使用 `[COPY_TBD: 场景描述]` 占位
- 但上线前必须替换完毕

## 八、文档与目录规范（推荐）

### 任务目录
每个功能一个任务目录：

```text
.agent/tasks/yyyy-MM-dd_[功能名]/
```

### 标准文档清单
- `方案设计.md`
- `执行计划.md`
- `审查报告.md`
- `测试报告.md`
- `P6_最终报告.md`
- `yyyy-MM-dd_[功能名]_最终报告.pdf`（推荐）

### 推荐规则
- 文档名稳定
- 目录按功能分开
- 不要把多个功能的阶段文档混在一起

## 九、P6 最容易出错的地方

### 1. 不要把 `planner 收口` 当成 `P6 完成`
必须拆开看：
- planner 收口完成
- P6 Markdown 完成
- P6 PDF 完成
- Git 合并完成
- 发布完成

### 2. 不要只产出 Markdown 却误说交付完成
如果项目要求 PDF 是最终交付物，必须明确：
- Markdown 只是中间稿
- PDF 才是最终交付物

### 3. 不要漏合并 / 漏部署
即使功能内部文档全齐：
- 没合并到主线，不算真正收尾
- 没部署上线，不算真正交付完成

## 十、推荐的编排器行为准则

适用于主编排 agent（例如 Dyna / Astra 这种角色）。

### 编排器负责：
- 派发角色
- 跟踪阶段
- 统一对外汇报
- 判断是否自动推进下一角色
- 判断是否满足发布条件

### 编排器不应：
- 混淆阶段状态
- 用“汇报已完成”替代“流程已推进”
- 默认一个 agent 包办全流程
- 在没有 P1/P2 的情况下直接放 builder 下场

### 编排器汇报格式建议
每次汇报尽量包含：
- 现在在哪一阶段
- 做完了什么
- 还缺什么
- 下一步自动会做什么

例如：

```text
标签聚合页：
- P3 已完成
- P4 主审 / 二审 / 三审中
- 当前无 blocking 结论
- 下一步：审查齐后自动进入 tester
```

## 十一、给其他 agent 的生成指令模板

你可以把下面这段直接丢给另一个 agent：

```md
请基于以下原则，为当前项目生成一套 OpenClaw 开发工作流文档：

1. 角色至少包含：planner、builder、reviewer、tester；若涉及 UI，再加入 designer
2. planner 负责 P1 方案设计、P2 执行计划、P6 最终收口
3. builder 只负责 P3，实现时可更新执行记录，但不得代写 P2
4. reviewer 负责 P4，支持主审/二审/三审并行
5. tester 负责 P5 测试验收
6. 多功能并行时，并行单位 = 不同功能；最多 4 路；超过则排队
7. 每个功能内部必须按 `planner -> builder -> reviewer -> tester -> planner` 流转
8. 在用户未叫停前，编排者必须自动推进到下一角色，不能停在阶段汇报上等待再次催促
9. 文档清单至少包含：方案设计.md、执行计划.md、审查报告.md、测试报告.md、P6_最终报告.md；若项目要求，最终交付物需导出 PDF
10. 输出要包含：
   - 角色分工
   - 阶段流转
   - 多功能并行策略
   - P6 收口规则
   - 文档目录规范
   - 适用于当前项目的定制约束

请将输出整理成一份适合直接写入项目 `AGENTS.md` / `.agent/OPENCLAW_WORKFLOW.md` / `docs/项目协作规范.md` 的正式文档。
```

---

## 十二、一句话总结

这套工作流的本质不是“多 agent 很炫”，而是：

**让不同角色各司其职，让每个功能按阶段稳定流转，让文档、代码、审查、测试、发布真正闭环。**
