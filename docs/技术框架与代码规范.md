# 技术框架与代码规范 v1

> 本文档定义博客项目的技术框架、目录结构、代码规范、命名约定。
> 所有 agent 和 contributor 必须遵循本文档。

---

## 1. 技术框架

### 1.1 技术栈

| 类别 | 技术 | 说明 |
|------|------|------|
| 框架 | Next.js 16 App Router | 全栈框架 |
| 语言 | TypeScript | 严格模式 |
| 样式 | Tailwind CSS 4 | 原子化 CSS |
| 认证 | NextAuth v4 | JWT session |
| 内容 | MDX | 文件驱动 |
| 部署 | Nginx + next start | 进程常驻 |

### 1.2 数据策略

- **V1**：文件驱动，`content/posts/*.mdx`
- **V2**：PostgreSQL + Prisma（按需引入）
- V1 禁止为未来扩展提前抽象数据库层

### 1.3 环境

- Node.js 22+
- 包管理：npm
- 构建：`npm run build`
- 开发：`npm run dev`
- 生产：`PORT=8080 npm run start`

## 4. 组件规范

### 4.1 组件分类

| 前缀 | 类型 | 位置 |
|------|------|------|
| `Site*` | 全局站点组件 | `components/` |
| `Post*` | 文章相关组件 | `components/` |
| `Auth*` | 认证相关组件 | `components/` |
| `Header*` | 头部组件 | `components/` |
| `Footer*` | 尾部组件 | `components/` |

### 4.2 组件原则

1. **页面组件用 Server Component**  
   除非需要交互（onClick、useState 等），否则用 `async function` Server Component。

2. **客户端组件必须注明 `"use client"`**  
   放在文件第一行或 import 之后。

3. **组件只做一件事**  
   不把多个不相关的 UI 块写进同一个组件。

4. **组件放在 `components/`**  
   页面放在 `app/`。

5. **禁止行内样式**  
   一律用 Tailwind class。

6. **组件文件不超过 200 行**  
   超过就拆分。

### 4.3 组件模板

```tsx
// "use client"  // 仅在需要客户端交互时加

import type { FC } from "react";

interface Props {
  title: string;
  // ...
}

export const ComponentName: FC<Props> = ({ title }) => {
  return (
    <div>
      {title}
    </div>
  );
};
```

## 5. MDX 文章规范

### 5.1 文件命名

```
yyyy-MM-dd_slug.mdx
```

示例：`2026-03-23_my-first-post.mdx`

### 5.2 Frontmatter Schema

```yaml
---
title: 文章标题
date: yyyy-MM-dd
slug: url-slug
excerpt: 文章摘要（用于列表页展示）
tags: [标签1, 标签2]
published: true      # true = 发布，false = 草稿
cover: /images/xxx.jpg  # 可选
---
```

### 5.3 约束

- `slug` 必须与文件名中的 slug 一致
- `date` 必须与文件创建日期一致
- `published: true` 才会在列表页显示
- 中文文章标题不加英文翻译
- 每篇文章必须有 `excerpt`

### 5.4 Schema 验证

`src/lib/posts.ts` 内置 Zod 运行时验证（`PostFrontmatterSchema`）。

所有 frontmatter 字段在读取时即时验证：
- 验证失败 → 文章标记为 **invalid**（不显示在公开页面）
- 验证通过 + `published: true` → **published**（公开）
- 验证通过 + `published: false` → **draft**（仅 Dashboard 可见）

| 字段 | 类型 | 必填 | 约束 |
|------|------|------|------|
| `title` | `string` | ✅ | 最小长度 1 |
| `date` | `string` | ✅ | 格式 `yyyy-MM-dd` |
| `slug` | `string` | ✅ | 最小长度 1 |
| `excerpt` | `string` | ✅ | 最小长度 1 |
| `tags` | `string[]` | ✅ | 默认为 `[]` |
| `published` | `boolean` | ✅ | **无默认值**，缺失即为 invalid |
| `cover` | `string` | ❌ | 可选 |

### 5.5 Dashboard 内容工作台

`/dashboard`（仅授权作者可访问）显示所有文章的状态：

- **已发布**：在 `/blog` 可见的文章
- **草稿**：`published: false` 的有效文章
- **待修复**：frontmatter 验证失败的文章（含具体错误信息）

`/dashboard` 通过 `getAllPostEntries()` 获取完整列表（包含 invalid），而 `/blog` 继续使用 `getPosts()`（仅返回 published）。

## 6. TypeScript 规范

### 6.1 基础规则

- 开启 `strict: true`
- 禁止 `any`，用 `unknown` 代替
- 所有 API 响应、组件 Props 必须有类型
- 函数返回值必须有类型注解

### 6.2 类型文件

| 文件 | 内容 |
|------|------|
| `src/types/next-auth.d.ts` | NextAuth 类型扩展 |
| `src/types/*.d.ts` | 全局共享类型 |

### 6.3 import 约定

```ts
// 顺序：React → 外部库 → 内部模块 → 相对路径
import { FC } from "react";
import Link from "next/link";
import { authOptions } from "@/lib/auth";
import { PostCard } from "@/components/post-card";
```

别名：
- `@/` → `src/`
- `@/components/` → `src/components/`
- `@/lib/` → `src/lib/`

## 7. Git 提交规范

### 7.1 提交类型

| 前缀 | 用途 |
|------|------|
| `feat:` | 新功能 |
| `fix:` | Bug 修复 |
| `docs:` | 文档改动 |
| `refactor:` | 重构（无功能变化） |
| `style:` | 格式、样式调整 |
| `test:` | 测试相关 |
| `chore:` | 构建、工具、依赖更新 |
| `perf:` | 性能优化 |

### 7.2 提交格式

```
<type>: <简短描述>

[可选的详细说明]
```

示例：

```
feat: add author whitelist for dashboard access

- Add GitHub user ID allowlist in auth config
- Restrict dashboard access to whitelisted users only
- Add environment variable for whitelist management
```

### 7.3 分支策略

- `main`：始终可部署
- 功能开发在 `main` 上直接小步提交
- 大功能可临时开分支，完成后合并

## 8. 代码风格规范

### 8.1 格式化

使用项目默认配置（ESLint + Prettier）。

### 8.2 通用规则

- **缩进**：2 空格
- **引号**：单引号（JSX 中用双引号）
- **分号**：加分号
- **空行**：组件之间空一行，函数之间空一行
- **行宽**：不超过 120 字符
- **注释**：关键逻辑加注释，不加无用注释

### 8.3 Next.js 特定规则

1. **不用 `new Image()`**  
   用 Next.js 的 `<Image>` 组件。

2. **路由组件默认 Server Component**  
   需要交互才加 `"use client"`。

3. **API 路由用 `route.ts`**  
   不用 `pages/api/`。

4. **`use client` 组件不引入 Server Component 直接数据获取**  
   客户端组件通过 props 接收数据。

### 8.4 Tailwind 规范

1. **用 Tailwind 的设计 token**  
   不自定义色板，先用现有颜色体系。

2. **响应式断点**  
   - `sm:` ≥ 640px
   - `md:` ≥ 768px
   - `lg:` ≥ 1024px
   - `xl:` ≥ 1280px

3. **深色模式**  
   用 `dark:` 前缀。基础样式不加 `dark:` 时表示两端通用。

4. **工具类顺序**  
   参照 Tailwind 推荐顺序：position → box-model → typography → visual → other。
