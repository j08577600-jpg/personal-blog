# 发布运营与基础分析

> V2 Workstream D 的安全子集实现。
> 只在现有 `posts` frontmatter 上落地，不抢统一内容模型定义权。

---

## 1. 当前已落地范围

### 发布 checklist

- 写作工作台右侧增加 `发布 checklist`
- 发布按钮不再只是切换 `published`
- 首次发布 / 更新发布前会弹出确认信息，展示：
  - 目标链接
  - 日期
  - 系列
  - 是否首页推荐
  - 更新记录
  - checklist 完成情况

### 文章运营字段（仅 `posts`）

支持以下 frontmatter：

```yaml
---
series: "V2 开发记录"
recommended: true
updatedAt: "2026-03-25"
updateNote: "补充了部署检查清单与回滚说明。"
---
```

说明：

- `series`：文章页展示同系列导航
- `recommended: true`：首页推荐区优先展示，博客列表排序提前
- `updatedAt` + `updateNote`：文章页展示最近更新信息，并进入搜索索引语料

### 最小运营面板

作者登录后可访问：

- `/dashboard/ops`

当前展示：

- 已发布文章数
- 首页推荐数
- 系列文章数
- 带更新记录文章数
- 当前分析策略说明
- 发布前检查与下一步判断标准

## 2. 当前明确没有做的事

这些内容故意后置，避免和 Workstream C 冲突：

- 不扩到 `projects` / `notes` / `reading-list` 的统一运营字段
- 不做完整 CMS / 运营后台
- 不做复杂推荐算法
- 不做评论系统
- 不直接接入重型分析平台

原因很简单：这些能力都依赖统一内容模型，当前仓库状态还不适合先把运营逻辑写死。

---

## 3. 分析策略

当前策略是：**先定义判断问题，再决定分析工具**。

现阶段建议观察：

- 来源（referrer）
- 热门文章
- 站内搜索词
- 推荐位点击是否更高
- 系列文章是否带来连续阅读
- 反馈邮件是否集中暴露问题

### 推荐接入顺序

1. 如果只想看页面访问、来源、热门内容：优先 `Plausible`
2. 如果想自托管：可以考虑 `Umami`
3. 若还没有稳定流量：先不接，继续通过搜索词、RSS、邮件反馈和内容更新判断

## 4. 反馈策略

当前阶段先保持简单：

- 站点页脚直接提供反馈邮箱
- 统一收集：勘误、合作、选题反馈、阅读反馈

默认邮箱：`hi@chenjilan.com`

如需后续升级，可以再判断是否需要：

- issue 模板
- 轻表单
- 读者问卷
- 评论系统

---

## 5. 发布前命令

本地或上线前至少执行：

```bash
npm run lint
npm run build
bash scripts/release-check.sh
```

如果是手工部署链路，再执行：

```bash
sudo systemctl restart personal-blog
```

## 6. 给下一阶段的交接说明

Workstream C 稳定后，D 可以继续扩展的方向：

- 把 `recommended / series / updatedAt / updateNote` 从 `posts` 抽到统一内容元数据层
- 决定 `projects` / `notes` / `reading-list` 是否共享同一运营字段口径
- 再评估是否需要站点级推荐编排或更正式的分析接入

在那之前，当前实现只负责建立最小秩序，不负责假装已经有完整运营系统。
