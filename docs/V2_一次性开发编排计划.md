# V2 一次性开发编排计划

> 日期：2026-03-25
> 状态：执行版
> 目标：在控制返工和回归风险的前提下，一次性完成 V2 全部功能开发，并最终以单次收口方式交付。

## 一、总原则

- 目标是最终顺利交付，不是表面上同时开很多任务。
- 采用“分主线开发、阶段错峰推进、最终统一收口”的方式。
- 明显改同一核心文件或同一核心数据结构的任务，不并行开发。
- 每条功能线独立完成 P1 / P2 / P3 / P4 / P5，最后统一进入 P6。
- 编排过程持续做关键节点汇报：P1/P2 完成、P3 完成、P4 完成、P5 完成、P6 完成。

## 二、V2 总任务树

### 主线 A：写作工作台升级

目标：提升写作、编辑、发布的高频使用体验。

范围：
- 自动保存
- 编辑恢复
- 发布确认
- frontmatter 编辑体验打磨
- 错误提示与防误操作优化
- 轻量版本差异 / 最近备份可读化

### 主线 B：认证扩展

目标：扩展登录入口，但不引入复杂账号系统。

范围：
- Google OAuth
- Microsoft OAuth
- 登录入口文案与交互整理
- email 白名单行为一致性验证

边界：
- 不做账号中心
- 不做复杂账号合并 UI
- 权限判断继续按 email 白名单

### 主线 C：内容模型统一 + 搜索增强

目标：把多内容类型变成一致、可维护、可搜索的一套结构。

范围：
- `posts / projects / notes / reading-list` 统一建模
- 统一索引字段 / 搜索字段 / 列表展示字段
- 搜索接入更多内容类型
- 类型 / 标签筛选
- 中文搜索召回与排序优化
- 清理现存搜索代码异味

### 主线 D：发布运营与基础反馈

目标：让站点从“能发”走向“可持续运营”。

范围：
- 发布 checklist
- 推荐位 / 置顶 / 系列能力
- 更新记录支持
- 最小可用数据反馈能力

## 三、依赖关系

### 可并行
- A（写作工作台）与 B（认证扩展）可先并行
- A/B 的 P4/P5 与 C 的 P1/P2 可并行
- D 的方案与前期调研可提前做，但实现不能抢在 C 稳定前落地

### 不建议并行
- A 与 C 同时大改 dashboard / 内容读写主链路
- C 与 D 同时大改内容结构与展示逻辑
- 多条线同时修改搜索索引、内容解析、公共类型定义

## 四、执行波次

### Wave 1：方案与执行计划
- designer-gemini：写作工作台 + 登录入口交互提案
- planner-codex：A / B / C / D 各自主方案设计与执行计划

### Wave 2：第一批核心开发
- builder-codex：A 写作工作台升级
- builder-codex：B 认证扩展
- 并行审查准备：提前定义 A/B 的验收标准与回归清单

### Wave 3：第一批审查测试 + 第二批开发
- reviewer-codex / reviewer-gemini / reviewer-minimax：A / B 三审并行
- tester-codex：A / B 测试验收
- builder-codex：C 内容模型统一 + 搜索增强

### Wave 4：运营与反馈能力落地
- 在 C 稳定后实现 D 发布运营与基础反馈
- 同时开始全站联调和回归预演

### Wave 5：统一收口
- 所有主线完成 P4/P5 后统一进入 P6
- 生成最终报告
- 合并分支
- 构建验证
- 部署
- 线上健康检查
- 对外汇报

## 五、并发控制

- 同时最多开启 4 路任务
- 常态推荐 3 路：
  - 1 路开发主任务
  - 1 路并行开发或方案任务
  - 1 路审查 / 测试 /收口任务
- 若某主线进入核心公共文件改动期，主动降并发，避免合流冲突放大

## 六、验收标准

### 功能层
- A / B / C / D 各自主线验收项全部达成

### 系统层
- 登录、作者权限、写作、保存、发布、搜索、内容展示能串成完整主链路
- 多 provider 登录不破坏白名单判定
- 多内容类型接入后，搜索和展示口径一致

### 发布层
- lint / build / 部署 / 健康检查通过
- 无阻塞审查项
- 无阻塞测试项

## 七、主要风险与压制策略

### 风险 1：写作工作台与内容模型同时改核心链路，导致返工
策略：A 与 C 错峰；A 先稳住写作链路，再推进 C。

### 风险 2：OAuth 增加 provider 后，作者权限行为不一致
策略：权限判断继续统一走 email 白名单；测试必须覆盖同邮箱多 provider。

### 风险 3：搜索扩容后索引结构与 UI 展示脱节
策略：C 统一定义索引字段与展示字段后，再做搜索增强。

### 风险 4：运营功能过早接入，建立在未稳定内容模型上
策略：D 后置，不抢 C 前面落地。

### 风险 5：任务周期过长导致上下文漂移
策略：严格按主线与波次推进，阶段完成即转下一角色，不挂空挡。

## 八、最终交付口径

V2 完成的定义不是“几个功能都做了”，而是：

- 主线 A / B / C / D 全部完成开发
- 审查与测试全部通过
- 统一分支完成最终合流
- 生产部署完成
- 线上健康验证完成
- 最终报告完成
