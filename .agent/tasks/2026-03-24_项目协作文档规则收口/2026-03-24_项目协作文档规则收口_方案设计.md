## 4. 现状问题分析

### 4.1 文件命名不一致

已完成的 11 个任务文件夹，按命名分三类：

| 类别 | 任务 | 文件名模式 | 是否符合规范 |
|------|------|------------|-------------|
| ✅ 标准 | MDXFrontmatter规范化、MDX内容系统、作者白名单、内容工作台 | `YYYY-MM-DD_功能_文档类型.md` | 是 |
| ⚠️ 不一致 | UI重设计_蓝白简约风、文章预览页、标签聚合页、站点地图_RSS | `文档类型.md`（无日期前缀） | 否 |
| ❌ 错误 | 内容工作台 | `task.md`（非标准文件名） | 否 |
| ❌ 缺失P1/P2 | 发布流程About与SEO | 只有 P4/P5，无方案设计/执行计划 | N/A |

**根目录残留**：`2026-03-23_UI重设计_蓝白简约风_方案设计.md` 落在 tasks 根目录而非对应文件夹内。

### 4.2 规范碎片化

规范分散在 3 处，表述存在差异：

| 文件 | 定位 | 问题 |
|------|------|------|
| `.agent/OPENCLAW_WORKFLOW.md` | 规范主文档 | 内容最全，但某些阶段规则描述模糊 |
| `.agent/roles/*.md` | 角色定义 | 与主文档有细微表述差异 |
| `AGENTS.md` 内嵌章节 | 项目规范摘要 | 缺失 P4/P5 回流规则，P6 规则不完整 |

### 4.3 P4/P5 回流规则缺失

现有规范未定义以下场景的处理方式：
- P4 审查发现方案设计存在根本缺陷，需要回退到 P1 重新设计
- P5 测试失败，发现是 P2 验收标准不够完善
- P4 必改项数量超过阈值，planner 需要介入范围调整

### 4.4 P6 md/pdf/merge/deploy 状态定义不清

| 状态 | 含义 | 当前定义 |
|------|------|---------|
| `md` | Markdown 源文件 | 是唯一编辑来源 |
| `pdf` | PDF 导出存档 | 可选/归档用，缺少明确说明 |
| `merge` | Git 合并状态 | 未定义合并时机和前置条件 |
| `deploy` | 生产部署状态 | 未区分灰度/全量/回滚条件 |

**P6 报告命名冲突**：部分任务同时存在 `P6_最终报告.md` 和 `yyyy-MM-dd_功能_最终报告.pdf`，两者内容是否等价未定义。

## 5. 方案设计

### 5.1 核心策略

**单一规范源原则**（Single Source of Truth）：`.agent/OPENCLAW_WORKFLOW.md` 是唯一的规范主文档，其他文件引用它但不包含完整规范。`AGENTS.md` 和角色定义文件只保留摘要和职责描述。

### 5.2 新增硬规则

#### 规则 1：任务文件夹命名

```
.agent/tasks/YYYY-MM-DD_功能名/
```

- 年-月-日四位数字开头
- 功能名不含下划线以外的特殊字符
- 同一功能只对应一个文件夹，不在 tasks 根目录残留独立文件

#### 规则 2：文档文件命名

每个文档必须包含完整日期前缀：

```
YYYY-MM-DD_功能名_文档类型.md
```

文档类型取值：
- `方案设计`
- `执行计划`
- `审查报告`
- `测试报告`
- `最终报告`

PDF 导出文件名与 md 文件名一致，后缀改为 `.pdf`。

#### 规则 3：P4/P5 回流规则

```
┌─────────────────────────────────────────────┐
│ P4 必改项 > 3 条，或存在「方案根本缺陷」    │
│ → 暂停 P4，回流 P1，重新设计               │
│ → planner-codex 介入，修订方案后重启流程   │
├─────────────────────────────────────────────┤
│ P5 失败项为「方案设计遗漏」导致             │
│ → 暂停 P5，回流 P1/P2，补充验收标准         │
│ → builder-codex 修复后重新审查             │
├─────────────────────────────────────────────┤
│ P4/P5 过程中发现范围蔓延                   │
│ → 通知 planner-codex，启动范围变更流程      │
│ → 新范围作为新任务独立处理                 │
└─────────────────────────────────────────────┘
```

**回流标记**：执行计划.md 中增加 `回流记录` 章节，记录每次回流的触发原因、决策和影响。

#### 规则 4：P6 md/pdf/merge/deploy 状态定义

| 状态 | 定义 | 触发条件 |
|------|------|---------|
| `md` | Markdown 源文件 | P6 开始时生成，存于任务文件夹 |
| `pdf` | 最终报告 PDF（归档用） | P6 Step 2 生成，与 md 等价 |
| `merge` | Git 合并完成 | P6 Step 3：`git push origin main` 成功 |
| `deploy` | 生产服务可用 | P6 Step 4：`pm2 restart` 或对应部署方式成功 |

**说明**：
- `最终报告.md` 和 `最终报告.pdf` 内容等价，md 是编辑源，pdf 是归档副本
- `P6_最终报告.md` 历史文件应统一重命名为 `yyyy-MM-dd_功能_最终报告.md`
- merge 前必须确认无阻塞性冲突；deploy 后才更新 DEVELOPMENT_PLAN.md
