## 二审结论（reviewer-gemini）

### 补充项

- [必改] `src/components/search-box.tsx:75` 仅做了"输入态 -> URL"的 debounce 同步，但缺少"URL -> 输入态"的回填 effect。当前 `query` 只在 `useState(initialQuery)` 首次渲染时取值，后续浏览器前进/后退或手动改 `?q=` 时，`urlQuery` 变化不会更新 `query`。这会直接导致执行计划中的验收项"浏览器前进/后退可恢复 URL 对应的搜索状态"失效，也会让输入框展示与地址栏脱节。
- [补充确认] 混合内容可读性整体方向是对的：`src/components/search-box.tsx:161` 用 section badge 区分"文章 / 项目"，`src/components/search-box.tsx:169` 改为使用 `href` 跳转，解决了 posts/projects 混排时的识别问题；结果文案也从"相关文章"调整为"相关内容"，方向正确。
- [补充建议] `src/lib/search.ts:64` 把项目结果卡片的标签优先切到 `tech`，会让项目卡片的标签含义与文章卡片不完全一致。它能提高技术关键词命中，但也让 UI 上的 tag 展示从"主题标签"变成"技术栈标签"。若这是有意决策，建议在主审或执行记录里明确；否则后续读代码的人容易误以为 `tags` 字段语义在所有类型中一致。

## 三审结论（reviewer-minimax）

### 安全风险项

- [✔] XSS 防护：query 值从 URL 读取后直接作为 React controlled input 的 value 渲染，React 自动转义，无 XSS 风险。
- [✔] URL 操作安全：`history.pushState(null, "", nextUrl)` 仅修改查询字符串部分，pathname 保持不变，不影响站内路由结构。
- [✔] 索引数据隔离：`scripts/build-search-index.ts` 复用 `getPosts()` / `getProjects()` 构建，bad-invalid-example 未进入索引，口径与站点公开内容一致。

### 中文表达项

- [✔] section badge：索引中 `section` 字段直接映射为"文章"（post）或"项目"（project），渲染时用 `entry.section` 输出，标签与内容类型一致。
- [✔] 标签口径：`mapProjectToSearchEntry` 优先取 `project.tech`，回退 `project.tags`，方向与二审建议一致（即 tech 作为技术栈标签），已明确在代码中体现，可接受。
- [✔] 结果文案："找到 N 条相关内容"适用于 posts + projects 混合结果集，语义准确；空查询提示语"输入关键词开始搜索文章和项目"完整告知搜索范围；空结果引导语提供博客与项目页跳转，UX 合理。

### 对前两审结论的最终确认

**二审阻塞项 — 确认已修复：**

- **urlQuery → query 回填**（`src/components/search-box.tsx`）：
  - `urlQuery` 通过 `useMemo(() => searchParams.get("q") ?? "", [searchParams])` 派生，每次 URL 变化后该值更新。
  - Effect 使用 `lastSyncedUrlQueryRef` 作为 guard，在 ref 与 urlQuery 不等时才执行同步，并在同步开头立即更新 ref，防止同一轮 render 中多次触发。
  - 同步使用 `setTimeout(..., 0)` 将 `setQuery` 推入下一 microtask，确保 ref 在 state update 前已更新到位，逻辑正确。
  - 该 Effect 覆盖：浏览器后退/前进、手动修改 `?q=`、从外部 URL 访问 `/search?q=...` 三条路径。

- **浏览器历史栈恢复**（`src/app/search/page.tsx`）：
  - Next.js App Router 对浏览器 popstate 事件有内置监听，浏览器后退/前进时自动触发 page 组件重新渲染并传入更新后的 `searchParams`，再透传给 `SearchBox(initialQuery)`，最终通过上述回填 Effect 同步到 query state。
  - Builder 已通过真实浏览器链路验证：`/search → /search?q=a → /search?q=b → 后退 → 前进`，history 长度正确增长，输入框与 URL 同步恢复。

### 边界情况复核

- [✔] 清空 query → URL 正确恢复为 `/search`（空字符串时 `params.delete("q")`）
- [✔] 刷新 `/search?q=...` 后状态保留（SSR 层透传 `searchParams.q` 作为 `initialQuery`）
- [✔] `q` 参数含空格/特殊字符：经 `normalizeSearchQuery` 规范化后与索引对齐，口径一致
- [✔] 快速连续输入：debounce 300ms 合并，仅最终值写 URL，无冗余历史栈项
- [✔] `bad-invalid-example` 未出现在 `public/search-index.json` 中
- [✔] 索引结构完整：`type / href / section / title / excerpt / tags / date / searchText` 均存在且正确

### 异议说明

**⚠️ 代码异味（低优先级，不阻塞上线）：**

`src/components/search-box.tsx` 第 83 行附近，在 `history.pushState` 后手动 `dispatchEvent(new PopStateEvent("popstate"))`。

**分析：**
- `history.pushState` 不会触发 native `popstate` 事件
- 当前组件没有任何 `addEventListener("popstate", ...)` 调用
- `lastSyncedUrlQueryRef` 在 pushState 后同步更新，状态由 useState 的 setQuery 更新链驱动（query → URL push → React 批处理 → 重新渲染），不依赖该 popstate 事件
- 浏览器后退/前进的路由变化由 Next.js App Router 内置处理，不走此手动事件

**结论：** 该行代码在当前实现中对功能无实际影响，疑似遗留的调试/兼容性代码。建议后续清理（删除该 dispatch），但**不阻塞本轮上线**。

### 复审结论

#### 1. URL/状态一致性边界 — ✅ 闭合

**urlQuery → query（URL 驱动的状态回填）：**

- `useMemo` 派生 `urlQuery = searchParams.get("q") ?? ""`，URL 变化后派生值更新
- Effect 用 `lastSyncedUrlQueryRef` 做等值 guard，防止同一 URL 值触发重复 setQuery
- `setTimeout(..., 0)` 将 `setQuery` 推入下一 microtask，确保 ref 已同步
- 入口覆盖：浏览器后退/前进、手动改 `?q=`、外部直接访问 `/search?q=`

**query → urlQuery（状态驱动的 URL 推送）：**

- debounce 300ms 合并快速输入
- `query.trim()` 为空时删除 `q` 参数，URL 正确恢复为 `/search`
- `window.history.pushState` 显式写浏览器历史栈，解决原 `router.push` 无法形成可回退栈的问题
- `lastSyncedUrlQueryRef` 在 pushState 前后同步更新，防止下一轮 effect 误判

**边界覆盖：**
| 场景 | URL 变化 | query 变化 | 结果 |
|------|---------|-----------|------|
| 外部访问 `/search?q=test` | ✅ | ✅ 回填 | 输入框显示 "test" |
| 输入 → debounce → URL 更新 | — | ✅ | URL 更新为 `?q=...` |
| 刷新 `/search?q=...` | ✅ | ✅ 回填 | 状态保留 |
| 清空输入框 | — | ✅ | URL 删除 `q` 参数 |
| 浏览器后退/前进 | ✅ | ✅ 回填 | 输入框 + 结果恢复 |

#### 2. 浏览器历史栈隐藏风险 — ✅ 无新增风险

- `history.pushState` 仅修改查询字符串，pathname 锁定在 `/search`，安全
- `lastSyncedUrlQueryRef` 同步更新时机正确，无跨轮次状态泄漏
- 浏览器前进/后退由 Next.js App Router 内置 popstate 处理，链路稳定
