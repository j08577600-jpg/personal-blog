# 搜索 V2 方案设计

> 日期：`2026-03-24`
> 任务：搜索 V2
> 负责人：planner-codex

## 目标

在现有 Fuse.js 客户端搜索基础上，为 `/search` 增加可分享、可恢复、可扩展的搜索体验：支持 URL 查询参数持久化、刷新不丢失搜索状态，并为后续纳入 projects / notes / reading-list 做好最小必要的数据结构升级，同时给出中文搜索体验的分层优化方案。

## 背景

搜索 V1 已完成基础能力：

- 构建期生成 `public/search-index.json`
- `/search` 页面通过 Fuse.js 在客户端搜索已发布 posts
- 搜索字段当前仅包含 `title / excerpt / tags / slug / date`

但 V1 已暴露出几个下一阶段问题：

1. 搜索词不支持 `?q=` 持久化，刷新或分享页面会丢失状态
2. 搜索索引仅覆盖 `posts`，与站点已存在的 `projects / notes / reading-list` 不一致
3. 中文搜索目前仅依赖 Fuse.js 默认模糊匹配，短词、别名、简繁、空格处理都比较弱
4. 现有索引生成脚本直接读取 frontmatter，绕过了 `src/lib/posts.ts` 的 Zod 校验，导致无效内容也可能进入索引

其中第 4 条不是体验优化，而是数据链路一致性问题。若不先修，V2 扩展索引范围时会把“每个内容类型各自一套校验逻辑”的问题继续放大。

## 现状结论

### 当前实现

- 页面：`src/app/search/page.tsx`
- 搜索组件：`src/components/search-box.tsx`
- 构建脚本：`scripts/build-search-index.mjs`
- 索引产物：`public/search-index.json`

### 已确认的问题

1. **前置阻塞问题：索引构建绕过内容校验**
   - `src/lib/posts.ts` 明确要求 `excerpt` 为必填且通过 Zod 校验后才算有效 post
   - `scripts/build-search-index.mjs` 直接用 `gray-matter` 读取 frontmatter，只检查 `published === true`
   - 结果：`content/posts/2026-03-23_bad-invalid-example.mdx` 虽然按站点内容规范属于无效文章，但仍进入 `public/search-index.json`
   - 影响：搜索结果与公共页面的数据口径不一致；扩展到多内容类型后会继续放大

2. **URL 不可恢复**
   - 搜索状态仅保存在组件内 `useState`
   - 刷新、前进后退、分享链接都无法恢复当前查询

3. **索引模型过于 posts-only**
   - 当前索引条目没有 `type / href / section` 等跨内容类型字段
   - 一旦纳入项目、笔记、阅读清单，前端结果渲染将不得不做兼容性重构

## 范围

### V2 必做

- 支持 `/search?q=关键词` 查询参数读写
- 首屏从 URL 恢复查询词，刷新后搜索状态不丢失
- 保持输入、URL、结果三者同步，且避免明显的路由抖动
- 搜索索引模型升级为“可容纳多内容类型”的统一结构
- **先纳入 `posts + projects` 到 V2 正式索引范围**
- 对 `notes + reading-list` 完成可行性评估、数据映射方案和后续接入预案，但默认不在本次正式上线范围内
- 修正索引生成链路，确保只索引“对外可见且结构有效”的内容
- 输出中文搜索体验的分层优化方案，并明确 V2 只落最小必要层

### V2 不做

- 不引入服务端搜索 API
- 不引入数据库、外部搜索服务、向量检索或 embedding
- 不做搜索建议、热门搜索、历史搜索
- 不做复杂筛选器（按标签、类型、时间组合过滤）
- 不做高亮分词片段、拼音搜索、同义词词库后台管理
- 不做一次性把所有内容类型都接入并上线；避免为了“未来扩展”提前做过重抽象

## 索引范围决策

### 为什么 V2 正式范围选 `posts + projects`

- `posts` 与 `projects` 都有完整的列表页 / 详情页 / excerpt 字段，搜索命中后的展示模型最接近
- `projects` 已有公开导航入口，用户对“项目也可搜索”预期明确
- 这两类内容都具备较稳定的标题、摘要、标签结构，接入成本低，结果页可统一展示

### 为什么 `notes + reading-list` 暂列后续项

- `notes` 没有 `excerpt` 字段，若直接纳入需定义摘要策略（正文截断、首段提取或无摘要样式）
- `reading-list` 的核心字段是 `type / author / source / status / rating / url`，与文章型内容差异更大
- 两类内容刚上线不久，用户搜索预期与结果样式还未稳定
- 先把统一索引结构打通，再决定是否全部开放，风险更低

### 评估结论

- **V2 上线范围**：`posts + projects`
- **V2 文档预留与映射定义**：`notes + reading-list`
- **V3 候选**：在真实使用反馈基础上，增量开放 `notes`，再评估 `reading-list`

## 统一索引模型

建议将索引条目从当前的 posts-only 结构升级为统一结构：

```ts
{
  type: "post" | "project" | "note" | "reading-item",
  slug: string,
  href: string,
  title: string,
  excerpt: string,
  tags: string[],
  date: string,
  section: string,
  searchText?: string
}
```

### 字段说明

- `type`：用于结果渲染和未来过滤
- `href`：直接跳转链接，避免前端按类型拼 URL
- `section`：展示文案，如“文章 / 项目 / 笔记 / 阅读”
- `excerpt`：统一保留；对没有摘要的类型允许后续通过截断正文生成
- `searchText`：保留为可选扩展字段，用于未来注入中文优化后的补充检索文本

### 设计取舍

- 不在 V2 引入复杂 schema 注册系统
- 仍由构建脚本集中拼出统一数组
- 每个内容类型只需提供“如何读数据”和“如何映射到统一索引结构”

这样足够支撑 2~4 种内容类型，而不会过度工程化。

## URL 状态方案

### 选型

采用 **App Router 原生 `searchParams` + 客户端 `useSearchParams` / `router.replace` 同步**。

### 交互规则

- 进入 `/search?q=agent` 时，输入框初始值为 `agent`
- 页面首屏在索引加载完成后自动执行搜索
- 用户输入后，debounce 结束时同时更新结果与 URL
- 使用 `router.replace` 而非 `push`，避免每次敲字都污染历史栈
- 当 query 为空时，URL 回退为 `/search`，不保留空 `?q=`
- 浏览器前进/后退时，输入框与结果按 URL 恢复

### 为什么不用本地存储

- 用户真正需要的是“可刷新、可分享、可回退”，不是“跨会话记忆”
- URL 是公开搜索页最自然的状态源
- localStorage 会引入多标签页状态不同步问题，且不可分享

## 中文搜索体验分层方案

### L0：现状保留

- Fuse.js 默认模糊匹配
- 检索字段：`title / excerpt / tags`

### L1：V2 必做的轻量优化

- 查询词做基础规范化：`trim`、压缩连续空格、统一全角/半角空格处理
- 索引文本做基础规范化：对标题、摘要、标签做同样的轻量处理
- 允许把 `title + tags` 赋予更高权重，弱化长摘要噪声
- 为中文短词场景适度收紧/校准 Fuse 配置，减少“完全无关结果也命中”

### L2：V3 候选优化

- 为每条内容增加额外 `searchText`，拼接标题别名、清洗后的正文片段、可能的中英混合关键词
- 对 notes / reading-list 生成更适合搜索的摘要文本
- 针对中文短 query 做更细的结果排序策略

### L3：暂不进入路线图的重方案

- 中文分词器接入
- 拼音检索
- 同义词词库
- 简繁转换检索
- 服务端/第三方搜索引擎迁移

### 分层结论

V2 只做 **L1 级轻量优化**。先把可恢复、可扩展和数据一致性解决掉，不把“中文搜索优化”误做成独立大型工程。

## 技术选择

### 方案 A：在现有静态索引 + Fuse.js 基础上增量升级（推荐）

- 保持 `public/search-index.json` 的静态索引模式
- 在构建期扩展为统一条目结构
- 搜索页继续在客户端加载索引并执行 Fuse 检索
- 用 URL 驱动查询状态

**优点：**
- 复用 V1 方案，改动集中，风险最低
- 符合当前项目“文件驱动、无数据库”的约束
- 能快速交付真正有感知的体验升级

**缺点：**
- 索引体积会继续增长
- 中文体验上限仍低于分词/服务端方案

### 方案 B：服务端 search route + 每次请求时过滤

**不选原因：**
- 与现阶段静态内容站定位不匹配
- 增加运行时复杂度，但未显著改善中文体验
- 只是把问题从前端移到服务端，没有解决数据模型和内容规范问题

### 方案 C：直接引入第三方搜索服务

**不选原因：**
- 对当前内容规模明显过重
- 增加外部依赖、同步链路和部署成本
- 现阶段用户价值不够高

## 风险与取舍

| 风险 | 影响 | 应对 |
|------|------|------|
| 继续沿用手写 frontmatter 解析，导致各内容类型口径不一致 | 搜索结果与页面展示不一致 | V2 前置修正索引构建链路，优先复用现有 lib 层或抽出统一的公开内容读取函数 |
| 一次纳入 4 类内容导致结果页样式和排序失控 | 交互复杂、验收变难 | 正式范围先上 `posts + projects`，其余先做映射评估 |
| URL 同步处理不当导致输入闪烁或重复搜索 | 体验退化 | 明确“输入态本地 state + debounce 后 replace URL + URL 变更回填”的单向规则 |
| 中文优化做得过重 | 范围膨胀 | 仅做 L1 轻量规范化与权重调整 |
| 搜索索引体积增长 | 首屏加载变慢 | V2 暂不拆分索引；待真实体积增长后再评估分片或懒加载 |

## V2 不该做成什么样

- 不要为了未来 10 种内容类型提前抽象成复杂插件系统
- 不要把“中文搜索优化”做成新一轮基础设施项目
- 不要一边扩索引、一边加筛选、一边改 UI 信息架构
- 不要在没有真实数据量压力时提前引入服务端或外部搜索

## 验收标准

- [ ] `/search?q=关键词` 首屏可正确回填输入框并展示结果
- [ ] 刷新 `/search?q=关键词` 后，搜索状态不丢失
- [ ] 清空查询后，URL 恢复为 `/search`
- [ ] 浏览器前进/后退可恢复 URL 对应的搜索状态
- [ ] 搜索索引仅包含对外公开且结构有效的内容，不再纳入无效 post
- [ ] V2 正式索引范围至少覆盖 `posts + projects`
- [ ] 搜索结果可区分内容类型并跳转到正确链接
- [ ] `notes + reading-list` 的映射与接入策略在文档中明确，但不强制本次上线
- [ ] 中文搜索体验至少完成 L1 级轻量优化，不引入额外后端依赖

## 结论

搜索 V2 采用“**静态索引延续 + URL 状态补齐 + 索引模型统一 + 轻量中文优化**”的增量方案。

执行优先级为：

1. 先修复索引构建绕过校验的问题
2. 再做 `?q=` 持久化和刷新恢复
3. 同步升级统一索引结构，并正式纳入 `projects`
4. 对 `notes + reading-list` 只完成映射与后续接入预案
5. 中文体验只做 L1 轻量优化，严格避免过度设计

满足以上条件后，可进入 P2 执行计划。
