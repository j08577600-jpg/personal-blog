# 站点地图 / RSS 方案设计

> 日期：2026-03-23
> 任务：站点地图 / RSS
> 负责人：planner-codex

## 目标

为博客提供 `sitemap.xml`（站点地图）和 RSS `feed.xml`（Feed 流），支持搜索引擎抓取和阅读器订阅。

## 背景

博客发布内容后需要：
1. **SEO**——搜索引擎通过站点地图了解站点结构，加快新文章收录
2. **RSS 订阅**——读者通过 RSS 阅读器（Feedly、Reeder 等）持续追踪更新，无需依赖社交平台

当前博客已有文章内容系统（`content/posts/*.mdx`）和公开页面路由（`/blog/[slug]`、`/about`），缺少这两项基础设施。

## 范围

### 做
- `sitemap.xml`：包含所有已发布文章页和重要静态页（`/`、`/about`、`/blog`）
- `feed.xml`：RSS 2.0 Feed，包含所有已发布文章的标题、链接、发布日期、摘要
- 技术方案选型与依赖评估

### 不做
- `robots.txt` 更新（可后续补充，不在本次范围）
- Atom / JSON Feed 格式（RSS 2.0 足够）
- 站点地图分页（Sitemap Index）（文章量级暂不需要）
- 动态 / 按需生成（非静态站点时再考虑）
- 已下线页面或归档内容的处理

## 技术选择

### 方案 A：`next-sitemap`（推荐）
- Next.js 官方推荐方案，`npm install next-sitemap`
- 配置 `next-sitemap.config.ts`，在 `postbuild` 时生成静态 `public/sitemap.xml` 和 `public/feed.xml`
- sitemap 由 next-sitemap 处理；RSS Feed 单独用脚本写入 `public/feed.xml`

**优点**：配置简单，CI/CD 友好，生产为静态文件，CDN 可直接缓存  
**缺点**：每次构建重新生成，中文 URL 需要 `lastmod` 等字段小心处理

### 方案 B：Next.js Route Handler（`/sitemap.xml` / `/feed.xml`）
- 用 `app/sitemap.ts` 和 `app/feed.xml/route.ts` 实现动态响应
- 每次请求实时读取文件系统生成 XML

**优点**：内容实时更新  
**缺点**：每次请求都读文件系统、解析 MDX；需要配置响应头 Cache-Control；生产环境性能不如静态文件

### 方案 C：混合
- `app/sitemap.ts` → 动态 sitemap（Next.js 内置）
- `app/feed.xml/route.ts` → 动态 RSS Feed

**优点**：内容实时  
**缺点**：同上，实时生成有性能开销；sitemap 还需要额外配置 `robots.ts`

### 决策

**采用方案 A：next-sitemap + 独立脚本生成 Feed**

理由：
- V1 简单优先，静态文件是最可靠、性能最优的方案
- next-sitemap 社区成熟，与 Next.js App Router 集成良好
- RSS Feed 生成只需复用已有的 `getPosts()` 逻辑，写入 `public/feed.xml`
- 未来如需实时内容，再迁移到 Route Handler

## 风险与取舍

| 风险 | 应对策略 |
|------|---------|
| next-sitemap 版本与 Next.js 16 不兼容 | 先检查兼容性列表；如有问题降级或改用 Route Handler |
| 中文 URL 编码导致 sitemap 校验失败 | 使用 `siteUrl` + 相对路径，确保 URL 符合 RFC 3986 |
| 新文章发布后 sitemap 延迟（仅在构建时更新） | 明确告知：需触发重新部署（或在 CI 中配置 `npm run build`）来更新 sitemap |
| feed.xml 生成脚本放在 CI 构建步骤失败 | 单独提供本地验证命令 `npm run generate:feed`，builder 执行后检查文件存在 |

## 结论

选定方案 A：以 `next-sitemap` 生成 sitemap，RSS Feed 复用 `getPosts()` 逻辑写入 `public/feed.xml`，均在 `postbuild` 阶段执行，输出静态文件。

**进入下一阶段判断**：方案已收敛，无重大未知风险，可进入执行计划阶段。
