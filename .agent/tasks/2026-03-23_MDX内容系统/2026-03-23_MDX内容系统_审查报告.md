for (const fileName of fileNames) {

### 5. 用户体验 — ⚠️ 两项建议项（非阻塞）

#### ✅ 做得好的

- 博客列表页设计简洁有力：标题用大字号 bold，摘要有节制，标签和元信息排版清晰
- 日期 + 阅读时长的元信息组合符合用户直觉
- 标签用 pill 样式展示，与现代博客 UI 惯例一致
- 文章详情页元信息与列表页风格统一，用户熟悉感强
- `published: false` 的文章不显示（由 `getPosts` 过滤），逻辑正确

#### ⚠️ 建议项 1：文章详情页缺少「返回」导航

点进一篇 article 后，浏览器左上角的返回按钮依赖用户习惯，但**没有显式的「返回博客」链接**。对于移动端用户或首次访问者，导航回退不够直觉。

建议在 `blog/[slug]/page.tsx` 的文章顶部加一行：

```tsx
<Link href="/blog" className="text-sm text-black/45 hover:text-black/80 dark:text-white/45 dark:hover:text-white/80 mb-6 inline-block">
  ← 返回博客
</Link>
```

#### ⚠️ 建议项 2：列表页无内容时缺少空状态 UI

`blog/page.tsx` 直接渲染 `posts.map(...)`，当 `getPosts()` 返回空数组时，页面会显示空白网格。虽然这在开发阶段不易察觉，但正式上线后如果文章被全部下架（或内容目录为空），用户看到的是空洞页面，无任何引导信息。

建议：

```tsx
{posts.length === 0 ? (
  <p className="text-black/45 dark:text-white/45">暂无文章。</p>
) : (
  posts.map((post) => <PostCard key={post.slug} post={post} />)
)}
```

---

### 1. 安全风险分析

| 威胁 | 结论 | 说明 |
|------|------|------|
| 路径遍历（Path Traversal） | ✅ 低风险 | `POSTS_DIR` 硬编码为 `process.cwd() + "content/posts"`，无用户输入进入文件路径拼接。`path.join` 在 `fileName` 为 `../etc/passwd` 时会正确拒绝超出 `POSTS_DIR` 的路径。 |
| 符号链接目录外跳转 | ⚠️ 理论风险 | 若 `content/posts/` 内存在指向上层目录的 symlink，`readFileSync` 会跟随跳转。个人博客场景下几乎不存在此攻击面，暂列低风险。 |
| 未捕获的同步异常 | ✅ 已覆盖 | `getPosts()` / `getPostBySlug()` 顶层 try/catch 覆盖 `readdirSync` 异常；`parsePost()` 内部 try/catch 覆盖单文件读取异常。任一环节失败均返回空数组或 null，不崩溃。 |

**安全结论**：无需修改，当前实现满足该规模项目的安全基线。

---

### 2. 健壮性审查

| 场景 | 当前行为 | 评估 |
|------|----------|------|
| `content/posts/` 目录不存在 | `readdirSync` 抛异常 → catch 返回 `[]` | ✅ 符合预期 |
| 单个 MDX 文件无读权限 | `readFileSync` 抛异常 → `parsePost` 返回 `null` → 列表中静默排除 | ✅ 符合预期，fail-gracefully |
| MDX 文件为空或 frontmatter 格式损坏 | `matter()` 可能抛异常 → `parsePost` 返回 `null` → 列表中排除 | ✅ 符合预期 |
| MDX 文件名含 `..` 或特殊字符 | `path.join(POSTS_DIR, "..")` 不会超出 `POSTS_DIR`，安全 | ✅ 符合预期 |
| `readdirSync` 与 `readFileSync` 之间文件被删除（race condition） | `readFileSync` 抛异常 → `parsePost` 返回 `null` | ✅ 符合预期 |
| `data.slug` 在 frontmatter 中缺失 | `parsePost` 回退为文件名；`getPostBySlug` 中 `data.slug === slug` 判断失败，回退路径被绕开 | ⚠️ 有隐患但不阻断（见下） |

**`data.slug` 缺失隐患（低优先级）**：若某 MDX 文件 frontmatter 缺少 `slug` 字段，`parsePost` 回退为文件名（带日期前缀），而 `getPostBySlug` 仍用 `data.slug` 比对，导致永远匹配不到。该文件的 `generateStaticParams` 会生成一个日期前缀的 slug，访问时永远 404。当前 3 篇文章 frontmatter 均完整，无实际影响；后续手动写 MDX 漏填 `slug` 会产生静默 404。建议在 `parsePost` 中对缺失 `data.slug` 的情况打一次 `console.warn`。

### 4. 额外风险点

| 风险 | 等级 | 说明 |
|------|------|------|
| **代码重复** | 中（维护性） | `getPostBySlug` 内重写了完整的 `matter` + `readingTime` 解析逻辑，与 `parsePost` 完全重复。若 frontmatter schema 变更，两处需同步修改，极易遗漏。当前功能正确，但长期维护有隐患。 |
| **性能无上限** | 低（当前可接受） | `getPostBySlug` 遍历所有 `.mdx` 文件做全量解析，无缓存，无 early return。10 篇以内无感；百篇级别应引入 LRU 缓存或 `Map<slug, Post>` 索引。 |
| **无自定义 404 页面** | 低（Next.js 默认够用） | `notFound()` 使用 Next.js 默认 404 页面。有条件可添加 `src/app/not-found.tsx` 与全站风格统一。 |
| **无文件大小限制** | 低（构建期可控） | `fs.readFileSync` 无上限，巨大 MDX 文件可能导致 OOM。当前构建环境可控；生产部署建议加入 `fs.statSync` + size limit 检查。 |
| **Next.js 缓存语义** | 低（主审已提及） | 主审已记录，建议后续显式 `cache()` 包裹 `getPosts()` 调用。 |
