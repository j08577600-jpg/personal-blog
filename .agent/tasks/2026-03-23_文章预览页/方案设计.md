# 文章预览页 — 方案设计

> 日期：2026-03-23
> 任务：Feature C · 文章预览页
> 负责人：planner-codex

---

## 目标

为已登录作者提供一个受保护的预览页面，可以渲染和查看 `published: false` 草稿文章的完整 MDX 内容。

---

## 背景

- 当前博客系统在 `content/posts/` 中支持 `published: false` 标记草稿
- 草稿文章**不会出现在公开 `/blog` 列表**（`getPosts()` 只返回 `published: true`）
- 已登录作者在 `/dashboard` 可以看到草稿列表，但**无法点击查看渲染效果**
- 缺少预览能力导致写作者无法验证 MDX 渲染、排版、标签是否符合预期

---

## 范围

### 做
- 新增 `/dashboard/preview/[slug]` 路由，受 auth 保护
- 仅允许已登录且在白名单内的用户访问
- 使用与 `/blog/[slug]` 相同的 `MDXRemote` 渲染引擎，保证预览与正式页一致
- 404 处理（slug 对应的文章不存在或为公开已发布文章时）
- 导航：页面顶部添加「返回控制台」和「返回博客」链接
- metadata 生成（title / description / og）

### 不做
- 不做发布/取消发布功能（留待后续功能）
- 不做版本历史对比
- 不做评论或审阅功能
- 不做公开草稿分享链接（即使 share token 也不做）

---

## 技术选择

### 方案 A：复用 `/blog/[slug]` 路由 + auth 守卫（不做）

在 `src/app/blog/[slug]/page.tsx` 加 auth 检查，合法用户可以看任意文章。

- 优点：无需新增文件
- 缺点：污染公开路由；需要额外逻辑区分预览模式；SEO 元数据不适用草稿

### 方案 B：独立预览路由 `/dashboard/preview/[slug]`（选定）

在 `src/app/dashboard/preview/[slug]/page.tsx` 创建独立页面。

- 优点：职责清晰，路由语义正确；与 dashboard auth 体系一致；不影响公开博客 SEO
- 缺点：需要复制部分渲染逻辑
- 应对：提取共用渲染组件（见下）

### 共用渲染组件

将 `src/app/blog/[slug]/page.tsx` 中的文章主体渲染抽取为 `components/article-renderer.tsx`，供预览页和正式页复用，保证两者完全一致。

---

## 风险与取舍

| 风险 | 应对策略 |
|------|---------|
| MDX 渲染逻辑重复维护 | 抽取 `ArticleRenderer` 组件，两页共用 |
| 草稿 slug 与已发布 slug 冲突 | 不做特殊处理 — slug 唯一性属于内容规范问题 |
| preview 页面 URL 被猜测泄露草稿内容 | 仅 auth 保护；V1 阶段可接受；未来可加 share token |
| builder 误将预览页逻辑用于公开路由 | 通过组件拆分和路由隔离约束 |

---

## 结论

采用方案 B：独立预览路由 `/dashboard/preview/[slug]`，抽取 `ArticleRenderer` 组件复用渲染逻辑。V1 做最小可用实现：auth 保护、MDX 渲染、基础元数据、返回导航。Scope 清晰，风险可控。
