# 监控与日志轮转收口 方案设计

> 日期：`2026-03-24`
> 任务：监控与日志轮转收口
> 负责人：planner-codex

## 目标

在不重复既有开发成果的前提下，为“监控与日志轮转”补齐最后一轮收口方案：把尚未完成的线上人工验证、部署后检查闭环、服务状态与日志验收动作整理成可执行清单，并明确哪些属于文档收口，哪些属于代码/脚本收口。

## 背景

`2026-03-24_监控与日志轮转` 已完成以下内容：
- `/api/health` 健康检查接口已交付并通过本地 lint/build
- `deploy/logrotate.conf`、`deploy/journald.conf补丁说明.md`、`deploy/DEPLOY_RUNBOOK.md` 已落地
- P4/P5 已确认无 blocking issue
- P6 已完成阶段性收口

但现有文档仍留下“部署后手动验证”缺口，具体表现为：
1. 线上是否已真正完成 `127.0.0.1:8080 -> Nginx 域名 -> systemd 服务 -> journald/logrotate` 全链路验证，现有材料里只有命令，没有闭环执行顺序。
2. 哪些动作必须在真实服务器执行、哪些动作可以在仓库里先补齐，没有被明确拆分。
3. 审查遗留的 3 个 non-blocking 项里，哪些只需文档澄清，哪些需要后续代码或脚本收口，没有单独归类。
4. 缺少“失败后如何恢复到稳定态”的统一说明，尤其是日志配置、服务重启、手动验证这三类场景。

因此，本次任务不是重复做监控能力本身，而是为已有成果补一份“可上线验收、可交接、可回退”的收口方案与执行计划。

## 范围

### 做
- 基于已有任务文档，梳理剩余收口项，不重复已完成开发范围
- 明确区分：
  - 真实线上环境必须执行的验证项
  - 仓库内可先完成的文档/脚本/配置收口项
- 为部署后检查建立闭环顺序：发布后先看什么、再验什么、异常时如何恢复
- 给出日志、健康检查、systemd 服务状态的可执行验收步骤
- 归类收口事项：哪些属于文档收口，哪些属于代码/脚本收口
- 为每项计划补充验收标准与恢复方式

### 不做
- 不新增监控功能，不重写 `/api/health`
- 不直接修改生产服务器、Nginx、systemd、journald、logrotate 实际配置
- 不进行真实部署、不跑线上命令、不生成运维执行结果
- 不重复撰写 P4/P5/P6 已确认通过的内容
- 不扩展到外部告警、Prometheus、Grafana、Sentry 等新能力

## 技术选择

### 方案 A：只补一页简单 checklist
- 优点：快，改动小
- 缺点：无法覆盖现有遗留项分类，也无法说明恢复路径
- 结论：不足以完成“收口”目标

### 方案 B：新增“收口任务”P1 + P2（推荐）
- 做法：在独立任务目录中，针对既有成果补充一份收口方案和执行计划
- 优点：
  - 不污染原任务文档，保留第一次交付记录
  - 能单独描述“剩余工作”而非重述已完成内容
  - 方便后续交由 builder/tester/运维按收口计划继续推进
- 缺点：需要额外维护一个任务目录
- 结论：最适合当前状态

### 收口分类策略

#### 文档收口
适合处理以下问题：
- 线上验证顺序不清晰
- 运行手册中缺少成功/失败判定标准
- 恢复步骤散落在不同文件中
- 审查遗留项需要先澄清适用范围

#### 代码/脚本收口
仅在以下情况需要进入后续开发：
- 需要新增脚本把多条手工命令收敛成统一验收入口
- 需要修正 `deploy/logrotate.conf` 的无效或重复配置
- 需要让 `release-check.sh` 或其他脚本纳入健康检查/服务状态检查闭环

结论：当前优先级应先做文档收口，再决定是否追加最小脚本收口；没有必要重新做功能开发。

## 风险与取舍

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 把已完成内容重新规划一遍，导致任务膨胀 | 中 | 中 | 明确以“剩余收口项”为唯一范围 |
| 线上验证步骤依赖真实服务器状态，仓库内无法完全确认 | 高 | 中 | 在计划中拆分“仓库内先完成”和“线上执行”两类动作 |
| 文档写得过泛，运维执行时仍需二次判断 | 中 | 高 | 所有线上步骤都补充命令、预期结果、失败恢复方式 |
| 审查遗留项长期悬空，后续被误认为已完成 | 中 | 中 | 把 3 个 non-blocking 项显式归类为文档收口或代码/脚本收口 |

取舍：
- 本次不做生产执行，接受“不能给出实际线上结果”的限制
- 但必须把线上执行前置条件、验收证据、失败恢复方式一次性定义清楚
- 优先保证可交接性，而不是追求自动化程度

## 结论

本次采用“独立收口任务”方案：
- 不重复原任务已完成的实现范围
- 聚焦剩余手动验证、部署后检查闭环、日志/健康/服务状态验收
- 将收口项拆为“线上必须执行”和“仓库可先完成”两类
- 将遗留项拆为“文档收口”和“代码/脚本收口”两类

满足以上条件后，才算为该功能建立真正可执行、可验收、可恢复的最终收口基线，并进入新的 P2 执行计划。
