# 阅读清单 / 碎片笔记 方案设计

> 日期：`2026-03-24`
> 任务：阅读清单 / 碎片笔记
> 负责人：planner-codex

## 目标

为博客新增两个内容品类：**阅读清单**（Reading List）和**碎片笔记**（Notes），均采用与现有 posts/projects 完全一致的文件驱动架构。

- 阅读清单：记录读过的书籍、文章、视频，包含阅读状态和评分
- 碎片笔记：短小、碎片化的随手记，不成文但有价值

## 背景

博客目前只有文章（posts）和项目（projects）两种内容形态。读者无法了解作者读什么、思考什么碎片化想法。增加这两个品类可以：

1. 丰富博客内容生态，增加读者粘性
2. 与 posts 形成互补（posts 是完整输出，notes/reading-list 是输入与思考碎片）
3. 完全复用现有 posts/projects 的文件驱动模式，无需引入数据库

## 范围

### 做
- `content/reading-list/*.mdx`：阅读清单 MDX 内容目录
- `content/notes/*.mdx`：碎片笔记 MDX 内容目录
- `src/lib/reading-list.ts`：阅读清单读取工具（getReadingList / getReadingItemBySlug）
- `src/lib/notes.ts`：碎片笔记读取工具（getNotes / getNoteBySlug）
- `/reading-list` 列表页（按 status 分组展示）
- `/reading-list/[slug]` 详情页
- `/notes` 列表页（按时间倒序）
- `/notes/[slug]` 详情页
- 首页侧边栏显示最新 3 条碎片笔记
- 示例 MDX 内容各 2 篇
- 站点地图 / RSS 同步（reading-list 和 notes 接入 sitemap 和 rss）

### 不做
- 后台管理页面（dashboard/reading-list、dashboard/notes）
- 用户阅读状态（用户端收藏、标记想读）
- 搜索索引纳入 reading-list 和 notes（等搜索功能稳定后 V2 扩展）
- 阅读进度追踪
- 自动书签导入
- 中文分词优化
- 非 MDX 的其他笔记格式

## 内容模型

### 阅读清单 MDX Frontmatter

```yaml
---
title: "书名 / 文章标题"
date: "2026-03-24"
slug: "book-slug"
type: "book"          # book | article | video | post
author: "作者名"       # 可选
source: "出版社 / 来源"  # 可选
url: "https://..."     # 可选，原文链接
status: "completed"   # want-to-read | reading | completed | abandoned
rating: 4             # 1-5，可选
tags: ["标签1", "标签2"]
published: true
---
```

**status 说明：**
- `want-to-read`：想读
- `reading`：在读
- `completed`：已读完
- `abandoned`：弃坑

**type 说明：**
- `book`：书籍
- `article`：文章
- `video`：视频
- `post`：博客文章

### 碎片笔记 MDX Frontmatter

```yaml
---
title: "笔记标题"
date: "2026-03-24"
slug: "note-slug"
tags: ["随想", "灵感"]
published: true
---
```

碎片笔记无 excerpt/author/source 等字段，内容即笔记本身，通常较短。

## 技术选择

### 方案 A：与 posts/projects 完全并行的文件驱动（推荐）

- `content/reading-list/*.mdx` 和 `content/notes/*.mdx` 独立目录
- 各自独立的 lib 工具函数（`reading-list.ts`、`notes.ts`）
- 各自独立的页面路由（`/reading-list`、`/notes`）
- 复用现有的 UI 组件（PostCard、ProjectCard 改写为 ReadingItemCard、NoteCard）
- sitemap 和 rss 扩展接入

**优点：**
- 架构一致，学习成本低
- 不污染 posts/projects 的类型系统
- 未来扩展灵活（可独立加 dashboard）

**缺点：**
- 代码略重复（lib 工具函数模式与 posts.ts 相同）
- 可接受，因为 V1 简单优先

### 方案 B：复用 posts 的泛型工具函数

- 扩展 `posts.ts` 为泛型 `getContentByDir(dir, schema)`
- 优点：减少代码重复
- **缺点**：增加泛型复杂度，破坏现有 posts.ts 的简洁性；reading-list 和 notes 的字段差异大，泛型收益低

**选定：方案 A**

## 风险与取舍

| 风险 | 应对 |
|------|------|
| lib 代码与 posts.ts 重复 | V1 可接受，后续可泛型重构 |
| 首页侧边栏加载 notes 增加构建时间 | notes V1 数量少，构建时直接读取文件，无性能问题 |
| sitemap/rss 扩展需改现有脚本 | 审查阶段注意覆盖，测试阶段验证 |
| notes 内容短，详情页可能显得单薄 | 详情页使用宽松排版，参考 Twitter/X 的卡片风格 |

## 验收标准

- [ ] `/reading-list` 页面可访问，返回 200
- [ ] `/reading-list/[slug]` 可访问，状态/评分/类型显示正确
- [ ] 阅读清单按 status 分组（want-to-read / reading / completed / abandoned）
- [ ] `/notes` 页面可访问，返回 200
- [ ] `/notes/[slug]` 可访问
- [ ] 首页侧边栏显示最近 3 条 published notes
- [ ] `npm run build` 时 sitemap 和 rss 包含 reading-list 和 notes
- [ ] `npm run lint` / `npm run build` 通过
- [ ] 现有功能（posts、projects、search）无回归

## 结论

采用完全并行的文件驱动架构，新增两个 MDX 内容目录和对应 lib/页面，不破坏现有 posts/projects 系统。V1 聚焦于内容呈现，无后台管理。
