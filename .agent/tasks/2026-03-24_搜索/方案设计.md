# 搜索 方案设计

> 日期：`2026-03-24`
> 任务：搜索
> 负责人：planner-codex

## 目标

为博客提供全文搜索能力，用户可在前端输入关键词检索文章标题、摘要、标签。

## 背景

当前博客文章通过标签页和直接浏览来发现内容，缺乏全文检索入口。随着文章数量增长，读者需要一个快速找到相关文章的方式。

## 范围

### 做
- 搜索页面 `/search`
- 构建期生成搜索索引（JSON），包含所有已发布文章的标题、摘要、标签、slug、日期
- 前端 Fuse.js 客户端模糊搜索（无需服务器）
- 搜索结果展示为卡片列表，点击跳转文章
- 页面顶部搜索框，支持实时搜索（debounce）

### 不做
- 搜索历史 / 热门搜索
- 后台搜索管理
- 全文内容索引（V1 只索引 title + excerpt + tags）
- 服务器端搜索 API
- 中文分词优化（V1 依赖 Fuse.js 逐字匹配）

## 技术选择

### 方案 A：构建期静态索引 + Fuse.js 客户端搜索（推荐）

- 构建时由脚本读取 `content/posts/*.mdx`，提取 frontmatter，输出 `public/search-index.json`
- 前端页面加载该 JSON，用 Fuse.js 做模糊匹配
- 无需数据库、无需服务端、无额外依赖（客户端搜索库 Fuse.js 约 7KB gzipped）

**优点**：部署简单、无冷启动延迟、无服务器成本、隐私友好
**缺点**：索引随构建产物，文章多时 JSON 体积线性增长（V1 文章量可控）

### 方案 B：API 路由 + 服务端过滤

- `src/app/api/search/route.ts` 接收 query 参数，读取文件过滤后返回
- 需要服务端运行时

**缺点**：增加 API 路由、部署复杂度、V1 不必要

### 方案 C：第三方搜索服务（Algolia、Typesense）

- 需要外部服务和 API key，增加运维负担

**缺点**：过度工程，不符合 V1 简单文件驱动原则

**选定：方案 A**

## 风险与取舍

| 风险 | 应对 |
|------|------|
| Fuse.js 中文逐字匹配体验不如分词 | V1 可接受，未来可换 zegodeen 或 ikAnalyzer |
| 文章数量超过 500 篇时 JSON 体积问题 | 构建时压缩 JSON；未来迁移服务端搜索 |
| 搜索框交互缺少 loading / 空状态 | 在执行计划中明确 UI 状态设计 |

## 结论

采用静态索引 + Fuse.js 客户端搜索方案，V1 极简实现，满足"快速找到文章"的刚需，无额外依赖和运维成本。

## 验收标准

- [ ] `/search` 页面可访问，返回 200
- [ ] 搜索索引 `public/search-index.json` 在 `npm run build` 时自动生成
- [ ] 搜索框输入关键词，实时（<300ms debounce）显示匹配结果
- [ ] 结果包含标题、摘要、标签，点击跳转正确文章
- [ ] 空关键词 / 无结果时有友好提示
- [ ] `npm run lint` / `npm run build` 通过
- [ ] 现有功能（博客列表、标签页、文章页）无回归
