# 文案修改提案

**任务角色**：designer-gemini
**日期**：2026-03-23
**状态**：设计提案 → 待 planner-codex 评审并输出执行计划

---

## 1. 提案总结

整体来看，现有站点的文案质量已经不错——克制、有结构感、不过度营销。但要真正契合「现代简约·蓝白·内容优先」的风格，需要在以下几个维度上做收敛：

| 维度 | 当前状态 | 目标 |
|------|----------|------|
| 语气 | 偏书面、有解释感 | 更加笃定、直接，像一个成熟作者在说话 |
| 层级标签 | 用「精选文章」「站点定位」等描述性标签 | 改用更功能性的标签，减少文学感 |
| CTA | 「查看文章」「关于本站」等 | 更加行为化、结果导向 |
| 元数据/空状态 | 较完整，但部分文字偏长 | 精简，信息密度提升 |
| 品牌感 | 「jay / 个人博客」 | 品牌感更强，像一个正式的产品 signature |

## 2. 逐页文案方向

### 2.1 首页（`src/app/page.tsx`）

**现状问题**：
- Hero 段落略显冗长（「用一个安静、清楚的空间，记录工程实践、项目进展和更锋利的思考。」）
- 「站点定位」侧边栏标题偏口语（「这个博客用来做什么」）
- CTA 链接「关于本站」「下一步计划」文学感过强
- 副标签「精选文章」偏编辑语气

**修改方向**：
- Hero 精简为更直接的陈述，去掉「安静」「清楚」等描述词，改为「专注工程实践与长期思考。」
- 侧栏标题改为「关于这里」或「本站说明」
- CTA 改为更功能性：「阅读文章」「了解更多」
- 精选区域改标题为「最近更新」或「最新」，去掉「精选」这种编辑腔
- 整体语气：从「向访客解释这是什么」转向「这是一个已经存在且有态度的空间」

---

### 2.2 博客列表（`src/app/blog/page.tsx`）

**现状问题**：
- 页面标题「记录构建、交付，以及更认真地思考问题。」偏长
- 空状态「暂无文章。」过于简单

**修改方向**：
- 页面标题精简为：「工程笔记与项目记录。」
- 空状态改为：「暂无文章，待更新。」
- 可考虑加一行辅助文字：「如有需要，可访问 GitHub 获取最新动态。」

---

### 2.3 文章详情（`src/app/blog/[slug]/page.tsx`）

**现状问题**：
- 「← 返回博客」链接过于简单，没有说明感
- metadata 中的 description 在首页已足够，详情页可适当精简

**修改方向**：
- 「← 返回博客」可改为：「← 返回列表」或「← 全部文章」
- 如有阅读时间预估，可加一行：「预计阅读 {readingTime}」

---

### 2.4 标签页（`src/app/blog/tags/page.tsx` & `[tag]/page.tsx`）

**现状问题**：
- 标签页标题「按标签浏览」偏描述性
- 标签详情页空状态「该标签下暂无已发布文章。」偏说明文

**修改方向**：
- 标签首页标题改为：「所有标签」或「标签索引」
- 空状态：「暂无已发布文章」改为：「此标签下暂无内容。」
- 标签详情页标题保持标签名 + 计数即可

---

### 2.5 About 页面（`src/app/about/page.tsx`）

**现状问题**：
- 页面标题「一个为清晰表达而建的个人站点。」偏长
- 「技术栈」和「内容方向」两个 section 的文案已经很好，可以微调
- CTA「← 查看所有文章」改为更直接的「← 浏览文章」

**修改方向**：
- 页面标题精简为：「为清晰表达而建。」或「关于这个站点」
- 技术栈 section 标题改为「技术选型」
- 整体保持克制，但去掉「会尽量保持」「不追求」等弱化词

---

### 2.6 Dashboard（`src/app/dashboard/page.tsx`）

**现状问题**：
- 标题「内容工作台」+ 用户名，语气偏硬
- StatCard 标签「已发布」「草稿」「待修复」已经很清晰
- 「写作提示」section 的文案有代码说明感，可以更友好
- 空状态「还没有文章。去 content/posts/ 创建第一篇 MDX 文件吧。」过于技术化

**修改方向**：
- 页面标题可改为：「作者后台 {用户名}」或「内容管理」
- 写作提示改为：「写作规范：文章存放于 `content/posts/`，文件名格式 `yyyy-MM-dd_slug.mdx`，`published: false` 为草稿，所有 frontmatter 字段必填。」
- 空状态更友好：「还没有文章。开始在 `content/posts/` 目录下创建你的第一篇 MDX 文件吧。」

---

### 2.7 登录页（`src/app/login/page.tsx`）

**现状问题**：
- 「登录后，逐步解锁后续写作与发布流程。」略长
- 副说明「第一版会把认证保持得足够轻...」过于解释

**修改方向**：
- 主标题改为：「登录以继续」或「登录访问作者后台」
- 副说明精简为：「使用 GitHub 账号登录，解锁私有写作与发布能力。」
- 或更极简：「通过 GitHub 登录，继续访问。」

---

### 2.8 未授权页（`src/app/unauthorized/page.tsx`）

**现状问题**：
- 「当前账号不在作者白名单内。」是描述性语言
- 建议文字「如需访问，请联系博主添加白名单。」略正式

**修改方向**：
- 主标题：「访问受限」或「需要授权」
- 说明：「当前账号暂无后台访问权限。请联系博主添加白名单。」
- 整体保持冷静、礼貌

---

### 2.9 导航栏（`src/components/site-header.tsx`）

**现状问题**：
- Logo「jay / 个人博客」偏朴素

**修改方向**：
- 改为：「jay / blog」或「jay.dev」
- 追求简洁和现代感

---

### 2.10 页脚（`src/components/site-footer.tsx`）

**现状问题**：
- 「记录工程、工具、项目与值得保留的判断。」和「基于 Next.js 构建，尽量克制，尽量长期可维护。」两句偏解释性

**修改方向**：
- 第一句保持
- 第二句改为：「Powered by Next.js」或直接用小字「Built with Next.js」
- 整体更简洁、现代

## 4. 风险与约束

1. **文案改动不等于 UI 改版**：本提案只涉及文字，不影响布局、颜色或组件结构。
2. **SEO 影响**：首页 description、博客页 description 等会随正文改动而变化，建议改动后重新提交 Google Search Console。
3. **Next.js Metadata API**：部分文案（如 `generateMetadata` 中的 description）在 Next.js 层定义，改动后需确认 build 正常。
4. **内容优先原则**：所有文案应服务于内容可读性，不要为了「简洁」而牺牲必要信息。
5. **中文语境**：当前站点面向中文读者，文案风格应保持中文母语感，避免翻译腔。
