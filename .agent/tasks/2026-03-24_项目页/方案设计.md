# 项目页 — 方案设计

> 日期：2026-03-24
> 任务：Feature 4 · 项目页
> 负责人：planner-codex

---

## 目标

为博客新增「项目」内容类型和公开页面，展示作者已完成或正在进行的项目，提供独立的项目详情页。与文章系统并行建设，保持相同的文件驱动模式和技术约束。

---

## 背景

- 当前博客仅有文章内容类型（`content/posts/*.mdx`）
- 作者（jay）有实际项目需要展示，但没有合适的位置
- V1 阶段无需数据库，所有内容继续走 MDX 文件
- 项目页与文章页在 UX 上保持一致风格，降低认知负担

---

## 范围

### 做
- 新增 `content/projects/*.mdx` 内容目录及 MDX 文件规范
- 新增 `src/lib/projects.ts` 数据读取层（与 `posts.ts` 同模式）
- 新增 `/projects` 列表页：展示所有已发布项目，支持按状态筛选（全部 / 进行中 / 已完成）
- 新增 `/projects/[slug]` 详情页：渲染 MDX 内容，含项目元信息（技术栈、外链、时间、状态）
- 新增 `src/components/project-card.tsx`：列表卡片组件
- 导航栏增加「项目」入口
- SEO / Open Graph / Twitter Card 支持

### 不做
- 不做项目标签聚合页（与文章标签体系解耦，V2 可扩展）
- 不做项目「收藏」或「喜欢」功能
- 不做项目搜索
- 不做草稿预览（V1 项目不做 `published: false` 预览，与文章预览功能独立）
- 不做项目评论或讨论

## 内容模型

### MDX Frontmatter Schema

```yaml
---
title: "项目名称"           # 必填，string
date: "2026-03-24"          # 必填，yyyy-MM-dd
slug: "project-name"        # 必填，string，URL slug
excerpt: "简短描述"         # 必填，string
published: true             # 必填，boolean
tags: ["React", "TypeScript"] # 可选，string[]
cover: "/projects/cover.jpg"  # 可选，项目封面图
status: "active"            # 可选，枚举: "active"|"completed"|"archived"，默认 "active"
featured: true               # 可选，boolean，默认 false（列表页置顶显示）
links:                      # 可选，对象
  github: "https://github.com/..."
  live: "https://example.com"
tech: ["Next.js", "TypeScript"] # 可选，string[]，技术栈标签
---
正文 MDX 内容...
```

### `status` 语义

| 值 | 含义 | 列表页处理 |
|----|------|-----------|
| `active` | 进行中/维护中 | 默认展示 |
| `completed` | 已完成/稳定版 | 默认展示 |
| `archived` | 归档/停止维护 | 列表页默认隐藏，点击「显示归档」后展示 |

### 字段设计原则

- 与 `PostFrontmatterSchema` 保持相同必填字段结构（`title/date/slug/excerpt/published`），降低学习成本
- `tags` 与文章标签共用 `getAllTags()`，不对项目标签单独建模（V1 简化）
- `links` 是扁平对象，避免嵌套过深
- `tech` 语义上与 `tags` 有重叠，但专用于展示「技术栈」——V1 合二为一都用 `tags` 显示，详情页的「技术栈」也读 `tags` 字段

---

## 技术选择

### 方案 A：项目与文章共用同一内容目录（不做）

将项目 MDX 文件放入 `content/posts/`，加 `type: project` frontmatter 区分。

- 优点：无需新增目录
- 缺点：内容类型混杂，路由判断复杂，SEO 元数据不一致

### 方案 B：独立 `content/projects/` 目录 + 独立 lib（选定）

新建 `content/projects/*.mdx` 和 `src/lib/projects.ts`，与 posts 完全并行。

- 优点：职责清晰，与 posts 完全解耦；frontmatter schema 可独立演进；路由结构对称
- 缺点：多一个内容目录
- 应对：posts.ts 模式直接复制，改名改路径即可，开发成本极低

---

## 风险与取舍

| 风险 | 应对策略 |
|------|---------|
| 项目与文章标签体系各自独立，用户困惑 | V1 简化为：项目详情页的「标签」也读 `tags`，不单独建模标签系统 |
| `/projects` 初期内容少，页面单薄 | V1 允许空状态提示；内容随时间自然积累 |
| 归档项目长期不维护 | V1 默认隐藏归档；详情页 URL 保持有效，不删除文件 |
| 项目详情页 MDX 渲染与文章页不一致 | V1 直接复用 `ArticleRenderer` 组件（抽取自文章预览页功能），保证完全一致 |
| 新增 `content/projects/` 目录后内容规范无强制校验 | V1 不引入 Zod 校验（与当前 posts 不同），风险可控；builder 需手动遵守 frontmatter 规范 |

---

## 结论

采用方案 B：独立 `content/projects/` 目录 + `projects.ts` lib 层。V1 最小可用：列表页（支持筛选）+ 详情页（MDX 渲染）+ ProjectCard 组件 + 导航入口。文件驱动，无 DB，模式与 posts 完全对齐，风险低、可演进。
