# 标签聚合页 方案设计

> 日期：`2026-03-23`
> 任务：标签聚合页
> 负责人：planner-codex

## 目标

实现博客标签聚合系统：提供标签索引页（展示所有标签及文章数）和标签详情页（展示该标签下所有已发布文章），支持按标签浏览和发现内容。

## 背景

博客已有文章标签体系（`tags: string[]` 存储于 MDX frontmatter），`PostCard` 组件也渲染了标签展示，但标签目前是纯展示性的，不可点击。

用户无法通过标签发现内容，标签体系的价值未充分释放。本次 V1 以最低成本补齐这一路径。

## 范围

### 做
- **标签索引页** `/blog/tags`：列出所有去重标签，每个标签显示关联文章数量，点击进入标签详情页
- **标签详情页** `/blog/tags/[tag]`：显示该标签下所有已发布文章，按日期倒序
- **标签链接**：更新 `PostCard` 组件，使标签变为可点击链接指向 `/blog/tags/[tag]`
- **SEO 元数据**：两个页面均生成 appropriate metadata（title、description、canonical）
- **`getAllTags()`**：新增 lib 函数，从已发布文章中聚合所有标签及其文章数

### 不做
- 标签云/可视化样式（留作后续 UI 增强）
- 标签管理页面（内容工作台已覆盖）
- 按标签筛选/搜索（无此需求）
- 标签别名/同义词
- 数据库层抽象（内容源只走 MDX 文件，V1 禁止）

## 技术选择

### 方案 A：在 `posts.ts` 中新增 `getAllTags()`（推荐）

**做法：**
- 新增 `getAllTags(): Array<{ tag: string; count: number }>`，读取所有已发布文章，提取并去重 tags，统计每标签文章数
- 标签索引页和详情页均为 **Server Component**（`async function`），直接调用 lib 函数
- URL 策略：`/blog/tags/[tag]`，tag 参数保留原始大小写（无需 toLowerCase），与 frontmatter 中的 tag 字符串精确匹配

**优势：**
- 最小改动，新增一个函数即可
- Server Component，无客户端 JS 负担
- 与现有 `getPosts()` 体系一致，共享 frontmatter 校验逻辑

**劣势：**
- 每次请求都读取文件系统（但 V1 体量下可忽略，后续可加 LRU 缓存）

### 方案 B：静态预生成标签数据

**做法：** 构建时将标签聚合结果写入 JSON 文件，运行时直接读取。

**劣势：** 增加构建复杂度，破坏内容实时性，V1 不值得。

### 方案 C：引入数据库层

**做法：** 将标签数据持久化到 SQLite/PostgreSQL。

**劣势：** 违背 V1 文件驱动原则，过度工程化。

**结论：采用方案 A**——最小化改动，与现有文件驱动架构一致，V1 足够了。

## 标签 URL 策略

- `tags` frontmatter 字段存储原始字符串（如 `"TypeScript"`、`"React 19"`）
- URL slug 使用原始字符串（如 `/blog/tags/TypeScript`、`/blog/tags/React 19`）
- 匹配逻辑：`post.tags.includes(tag)`（精确匹配，大小写敏感）
- 不做 slugify/normalize，避免大小写/空格歧义

## 标签链接更新 PostCard

现有 `PostCard` 渲染标签为 `<span>`，改为 `<Link>` 指向 `/blog/tags/[tag]`。

- 需要 `"use client"`（Link 是客户端组件，但可内联）
- 标签样式不变，仅加 `href`

## 风险与取舍

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 标签含特殊字符导致 URL 失效 | 低 | 中 | URL 编码处理，Next.js Link 自动编码 |
| 标签数量多时索引页加载慢 | 低 | 低 | V1 体量下可忽略，后续加缓存 |
| PostCard 升级为客户端组件影响性能 | 低 | 低 | Link 内联无需整组件变 client |
| 多语言/中英混用标签不一致 | 低 | 低 | MDX 规范要求作者自控，V1 不处理 |

## 结论

- **方案 A**（`getAllTags()` + 两个 Server Component 页面 + PostCard 标签链接化）是最小化、最安全的 V1 实现
- 范围明确：不做高级样式、不做数据库、不做标签管理
- 技术上完全可行，进入执行计划阶段
