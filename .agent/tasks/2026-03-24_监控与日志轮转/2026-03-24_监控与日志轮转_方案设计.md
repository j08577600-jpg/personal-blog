# 监控与日志轮转 方案设计

> 日期：`2026-03-24`
> 任务：监控与日志轮转
> 负责人：planner-codex

## 目标

在单台服务器上建立轻量级监控体系：健康检查接口、日志轮转、systemd journal 管理，提升故障发现速度和可运维性。

## 背景

当前部署链路（Nginx → Next.js on systemd）已稳定运行，但存在以下可运维性问题：

1. **无健康检查接口**：`/api/health` 不存在，`release-check.sh` 依赖 curl `http://127.0.0.1:8080` 判断存活，Next.js 默认页面不返回结构化健康状态。
2. **日志无轮转**：systemd journal 无限增长，Next.js stdout/stderr 由 journald 管理，无独立轮转策略。
3. **可观测性不足**：故障排查依赖 `journalctl`，没有结构化的健康端点可供外部探测（如 UptimeRobot、Cf Health Check）。

## 范围

### 做
- 新增 `/api/health` 接口：返回 `{ ok, timestamp, version }`，`ok=false` 时 HTTP 500
- 配置 `logrotate` 对 systemd-journald 的 Next.js 相关日志做转储压缩
- 在 `DEPLOY_RUNBOOK.md` 中补充监控自查清单
- 添加 PM2/process manager？不做。当前 systemd 足够。

### 不做
- 外部监控服务接入（UptimeRobot、Cf Health Check 配置由运维自行决定）
- 指标收集（Prometheus / Grafana 等）
- 告警到钉钉/Slack
- 多服务器健康探测
- Next.js Sentry / error tracking（Feature 独立规划）
- systemd service 文件改造（保持简单）

## 技术选择

### 健康检查接口

- **方案 A（推荐）：新增 `src/app/api/health/route.ts`**
  - 优点：Next.js App Router 标准 API Route，无额外依赖，返回结构化 JSON
  - 缺点：需要 commit/build 才能生效（但 release-check.sh 本身就需要 rebuild）
  - 风险：极低

- **方案 B：使用 Next.js 内置 `/ready` 或自定义 server**
  - Next.js 16 没有开箱即用的健康检查端点
  - 需要自定义 server，放弃 `next start`，引入不必要的复杂度

结论：采用方案 A，5 行代码实现。

### 日志轮转

- **方案 A（推荐）：`logrotate` 配置 systemd journal**
  - Next.js stdout/stderr 由 systemd journal 管理（journald）
  - 通过 `systemd-cat` 或直接 systemd unit 的 `StandardOutput=journal` + `StandardError=journal` 写入 journal
  - `logrotate` 配置 `/var/log/journal/` 轮转（journald 持久化模式）或 `/run/log/journal/`（临时模式，本项目情况）
  - 优点：不改 systemd service，保持 journald 原样，轮转策略集中管理
  - 缺点：需要 `sudo` 权限写入 `/etc/logrotate.d/`

- **方案 B：journald 自身配置 `SystemMaxUse` + `MaxRetentionSec`**
  - 在 `/etc/systemd/journald.conf` 中配置 journal 大小和保留时间
  - 优点：无需额外工具，原生支持
  - 缺点：粒度粗，只能按大小/时间，不支持压缩和独立删除策略

- **方案 C：PM2 / 进程管理器日志管理**
  - 放弃 systemd，改用 PM2
  - 风险：引入不必要的进程管理器，systemd 已被验证可靠

结论：采用方案 A + 方案 B 结合。以 `logrotate` 为主，journald 配置兜底。

## 风险与取舍

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| logrotate 配置错误导致日志丢失 | 低 | 中 | 测试阶段 dry-run 验证轮转逻辑 |
| health endpoint 影响性能 | 极低 | 低 | 纯内存判断，无 IO |
| journald 未开启持久化（默认 tmpfs）| 中 | 低 | 告知用户 journal 重启后丢失，建议开启持久化 |
| logrotate cron 漏跑导致磁盘满 | 低 | 高 | journald.conf 兜底 SystemMaxUse |

## 结论

本次方案以最小侵入为原则：
- `/api/health` 5 行代码，不引入任何依赖
- `logrotate` 配置文件 + journald.conf 调优，双重保险
- `DEPLOY_RUNBOOK.md` 补充监控自查，让运维有据可查
