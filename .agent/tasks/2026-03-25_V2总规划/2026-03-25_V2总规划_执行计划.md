## 已知约束

- 内容系统继续以文件驱动为真源，不引入数据库层
- 作者权限继续按 email 白名单判断，不引入复杂账号系统
- 无 P1/P2 不进入开发，无审查通过不进入测试，无测试通过不进入发布
- 多功能可并行，但明显修改同一核心文件或同一核心数据结构的任务不并行
- 总体并发上限 4 路，常态推荐 3 路
- 最终发布前必须完成 `npm run lint`、`npm run build` 以及发布/健康检查链路

## 主线与共享区域映射

### A：写作工作台升级
- 主要区域：`src/app/dashboard/*`、作者态 API、作者态内容写入工具、写作表单组件
- 避让区域：统一内容模型、搜索索引脚本

### B：认证扩展
- 主要区域：`src/lib/auth.ts`、NextAuth 配置、登录页/授权页、provider 环境变量
- 避让区域：内容读取、搜索、运营编排逻辑

### C：内容模型统一 + 搜索增强
- 主要区域：`src/lib/*` 公共内容读取、schema/mapper、`scripts/build-search-index.*`、搜索页与结果组件、跨内容类型类型定义
- 避让区域：A 的作者态写作交互实现期、D 的运营字段具体落地期

### D：发布运营 / 运维 / 基础分析
- 主要区域：发布 checklist、首页内容编排、系列/置顶/推荐字段消费、最小分析接入、运维文档或轻量脚本
- 依赖区域：C 统一内容字段、A 写作状态与发布动作

## 中断恢复方式

如果总控流程在任何波次中断，下次恢复按以下顺序判断：

1. 先确认四条主线各自停在哪个阶段（P1/P2/P3/P4/P5/P6）
2. 若 A/B 尚未完成核心实现，不启动 C 的公共层改造
3. 若 C 未稳定，不启动 D 的正式实现
4. 若任何主线停在 P4/P5 且存在阻塞问题，优先处理回流，不直接推进 P6
5. 恢复时先更新本计划中的进度、决策日志和回流记录，再继续派发下一角色

## 各主线执行建议

### A 主线执行建议
- 先做保存状态和恢复策略，再做发布确认与差异可读化
- 避免在 A 内顺手抽象“通用内容编辑器”
- 审查重点放在：写入安全、恢复策略、交互反馈回归

### B 主线执行建议
- 先统一 auth 层 email 判断，再接 provider，再收口登录 UI
- 测试重点放在：同邮箱跨 provider、一致授权、非白名单拒绝
- 不要把 provider 差异扩散到 dashboard 页面判断层

### C 主线执行建议
- 先统一公开读取和索引契约，再做搜索筛选与排序优化
- 优先消除“绕过 schema 校验直接入索引”的问题
- notes / reading-list 可先映射到统一结构，但展示差异仍可保留

### D 主线执行建议
- 先选最小可用运营能力：推荐位/置顶/系列至少落一组
- 先定义分析指标，再选轻量实现，不反过来被工具牵着走
- checklist 尽量贴近真实发布流程，而不是只写文档

## 并发控制

- 常态最多 3 路并行：
  - 1 路开发主任务
  - 1 路低冲突并行任务或方案任务
  - 1 路审查/测试/收口任务
- 峰值最多 4 路，但仅限共享文件未进入高冲突期
- 以下情况主动降并发：
  - C 正在改公共内容模型或搜索索引脚本
  - A 正在改 dashboard 核心写作表单和写入链路
  - 任何主线进入跨目录大规模重命名或类型重构
