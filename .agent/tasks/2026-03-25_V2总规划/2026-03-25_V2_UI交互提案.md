# V2 UI / 交互提案

基于当前站点的蓝白极简风格、圆角卡片、细边框、弱阴影与安静的信息层级，V2 不建议重做视觉语言，重点是把高频作者动作、登录决策和多类型搜索变得更顺手、更少误操作。以下方案以“保留现有气质，增强结构与反馈”为原则。

---

## 1. 写作工作台升级

### 1.1 目标

把当前“单页表单 + 手动保存”的工作台，升级为一个更稳定的作者工作区：

- 更容易开始写
- 更不容易丢内容
- 更容易理解发布状态
- 更容易处理 slug / 日期 / frontmatter 变更带来的影响
- 不把界面做成重 CMS

### 1.2 关键用户流

#### Flow A：进入工作台 -> 继续最近草稿

建议把 `src/app/dashboard/write/page.tsx` 从“纯列表”升级为“两段式入口”：

1. 顶部保留当前标题区和“新建草稿”主按钮
2. 标题区下方新增一个“继续写作”卡片
3. 默认展示最近编辑的 1~3 篇内容，按最近修改时间排序
4. 每条显示：标题、状态、最后编辑时间、内容类型、是否有未发布改动
5. 点击整卡进入编辑页；右侧提供轻操作：`继续编辑` / `预览` / `复制链接`

这样首页不是只有文件列表，而是先回答“我现在最可能想做什么”。

#### Flow B：新建草稿 -> 最小必填 -> 进入正文

当前新建页仍偏“先填完整 frontmatter，再开始写”。V2 建议改成：

1. 新建时先展示最小字段：`标题`、`内容类型`、`草稿状态默认开启`
2. 输入标题后自动生成 slug 草案，但不强制马上确认
3. 日期默认当天，隐藏在“发布信息”区，不占第一屏注意力
4. 创建成功后直接进入正文编辑视图
5. 其他 frontmatter 放进可折叠的“文章设置”面板

原则：先进入写作，再补元信息。

#### Flow C：编辑中 -> 自动保存 / 离开保护 / 恢复

这是写作工作台最该补的一段。

建议增加三层保护：

1. `自动保存`
   - 正文或 frontmatter 变更后延迟 3~5 秒自动保存
   - 右侧操作区把主状态从按钮型文案改成状态型文案：`已保存` / `保存中...` / `保存失败` / `存在冲突`
2. `本地恢复草稿`
   - 保存前的输入状态同步缓存到 `localStorage`
   - 若服务端保存失败、网络中断或刷新中断，重新进入同 slug 页面时提示“检测到本地未恢复内容”
   - 恢复方式只给两个动作：`恢复本地内容` / `忽略`
3. `离开页保护`
   - 当存在未保存改动或保存请求尚未完成时，离开页面前给浏览器原生拦截提示

不要做复杂版本编辑器，先确保“不丢字”。

#### Flow D：发布 / 取消发布

当前 `publish-toggle` 是和保存同一链路，但交互还不够明确。建议：

1. 右侧状态卡把“保存”和“发布”拆成两个层级
2. `保存` 是日常动作，主按钮始终固定
3. `发布` 是次级但高风险动作，需要二次确认弹层
4. 确认弹层要展示：
   - 标题
   - slug / 最终 URL
   - 日期
   - 当前是首次发布还是更新发布
5. 若文章缺少关键字段，发布按钮直接禁用，并在按钮下方列出缺失项
6. 取消发布也做轻确认，但语气弱于首次发布

重点不是多一步，而是让作者知道“这次发布到底会发生什么”。

### 1.3 页面结构建议

#### 写作页布局

当前 `src/components/writer/workbench-form.tsx` 已是“主编辑区 + 右侧栏”。这个方向对，建议进一步明确为三段：

1. `顶部文档条`
   - 左侧：标题 / 草稿状态 / 内容类型
   - 中部：保存状态、最后保存时间
   - 右侧：预览、发布、更多操作
2. `主编辑区`
   - 默认第一屏优先展示正文编辑器
   - frontmatter 不要占据正文前面的大块空间
3. `右侧信息栏`
   - 文档状态
   - 发布信息
   - 文章设置
   - 恢复与历史入口

对于桌面端，正文区权重应明显高于元数据区；对于移动端，右栏折叠到底部抽屉或分节卡片，不要强行双栏。

#### Frontmatter 改造

`src/components/writer/frontmatter-fields.tsx` 目前是平铺字段。V2 建议改成分组：

- `基础信息`：标题、摘要
- `URL 与时间`：slug、日期
- `组织信息`：标签、内容类型、系列/推荐位（如果 D 线会接）
- `展示资源`：封面图

每组卡片标题下加一句非常短的提示，不要写帮助文档式大段说明。

其中：

- `slug` 字段旁增加“从标题重新生成”轻按钮
- `tags` 改为 token/tag 输入体验，不再只用逗号文本框
- `date` 对作者是业务字段，不是技术字段，优先显示成人类可读格式，同时提交时仍写标准值
- `excerpt` 增加字数建议，如 60~140 字

#### 正文编辑器改造

`src/components/writer/body-editor.tsx` 现在是单个深色 textarea，功能够用但写作氛围较粗糙。建议：

- 保留纯文本/Markdown 路线，不引入重型富文本
- 增加顶部轻工具条：
  - 插入标题
  - 插入链接
  - 插入代码块
  - 插入引用
  - 插入图片模板
- 工具条只做“插入模板”，不做复杂格式同步状态
- 编辑区旁边可选显示“Markdown 速查”抽屉，而不是长期占空间
- 长文编辑时增加“专注模式”：隐藏右栏，仅保留顶部状态条

视觉上保留深色编辑区，但建议把背景从纯黑改成更柔和的深蓝灰，和全站主题更一致，减少割裂。

### 1.4 组件变化建议

针对现有实现，建议新增/调整以下组件：

- `DocumentStatusBar`
  - 展示：草稿/已发布、自动保存状态、最后保存时间、冲突状态
- `AutosaveIndicator`
  - 统一处理 `idle/saving/saved/error/conflict`
- `DiscardOrRestoreDraftDialog`
  - 进入页面发现本地缓存时显示
- `PublishConfirmDialog`
  - 发布前确认
- `TagInput`
  - 替代逗号字符串输入
- `EditorToolbar`
  - 仅插入 Markdown 模板
- `WorkbenchSidebar`
  - 把当前散落在右栏的状态、限制、操作统一收纳

### 1.5 边界与异常处理

必须明确处理这些情况：

- `slug` 被改后，旧路由与新路由的跳转反馈要明确，避免作者以为内容丢失
- 自动保存时遇到 mtime 冲突，不能只报红字；要明确说明：`文件已被其他修改覆盖，需要手动决定保留哪一版`
- 首次创建但标题为空时，不要生成难看的占位 slug 写入文件系统
- 新建草稿默认不发布，这一规则继续保留
- 自动保存失败后，手动保存按钮要变成更显眼的补救动作
- 发布成功后，建议提供两个紧邻动作：`查看线上页面` / `继续编辑`

### 1.6 明确 Do / Don't

**Do**

- 优先正文写作流
- 优先保存状态可见性
- 优先误操作保护
- 优先恢复能力
- 让作者知道系统当前状态

**Don't**

- 不做 Notion 式块编辑器
- 不做复杂多人协作 UI
- 不做重历史版本时间线
- 不在第一屏堆所有 frontmatter 字段
- 不把发布动作伪装成普通保存

## 2. 登录入口 / Provider 选择

### 2.1 目标

V2 扩展到 GitHub / Google / Microsoft 后，登录页不该只是“多三个按钮”，而应清楚回答：

- 这是作者入口，不是公开用户系统
- 可以用哪个账号进
- 为什么可能登录成功但仍无权限
- 多 provider 权限判断仍以 email 白名单为准

### 2.2 关键用户流

#### Flow A：从站点头部进入登录页

当前 `src/components/header-auth-controls.tsx` 直接写“GitHub 登录”。V2 建议：

- 未登录状态改成更中性的 `作者登录`
- hover 或点击后进入 `/login`
- 不在 header 直接展开 provider 列表，避免头部噪音

这是因为登录是低频动作，header 更适合作为入口，不适合作为决策面板。

#### Flow B：登录页选择 provider

`src/app/login/page.tsx` 建议升级为三层结构：

1. `顶层说明`
   - 标题：作者入口
   - 副文案：支持 GitHub / Google / Microsoft 登录，权限仍按授权邮箱判断
2. `Provider 列表`
   - 三个按钮并列或纵向堆叠
   - 每个按钮保持统一尺寸与层级
   - 文案统一：`使用 GitHub 登录` / `使用 Google 登录` / `使用 Microsoft 登录`
3. `说明区`
   - 非白名单邮箱：可完成认证，但会进入无权限页
   - 建议使用与白名单一致的邮箱登录
   - 若多个 provider 对应同一邮箱，权限结果应一致

设计上不要把 GitHub 做成强主按钮、其他两个像附属入口。V2 后三个 provider 地位应平齐。

#### Flow C：登录成功但无权限

当前已有 `/unauthorized`，这里建议把文案更具体：

- 明确说明：认证成功，但当前邮箱未被授予作者权限
- 展示当前识别到的邮箱
- 给出两个动作：
  - `返回首页`
  - `更换账号重新登录`

这样能显著减少“是不是系统坏了”的误解。

### 2.3 页面与组件建议

#### 登录页布局

保留当前大卡片结构，但增加信息节奏：

- 第一屏中部：登录卡片
- 卡片内部：
  - 顶部小标签：`作者入口`
  - 大标题
  - 简短解释
  - provider 按钮组
  - 分隔线
  - 权限说明

可以继续沿用全站圆角、细边框、浅阴影，不需要单独做品牌页视觉。

#### Provider 按钮样式

建议抽象成统一组件，例如 `AuthProviderButton`：

- 左侧 provider logo
- 中间文案
- 右侧轻箭头或外部跳转感应图标都可以，但尽量克制
- hover 只做边框/背景轻变化，不做强动画
- 错误状态支持按钮下方行内提示

#### Header 登录状态

登录后，`src/components/header-auth-controls.tsx` 不要只显示用户名文本。建议：

- 桌面端：显示简短身份 chip，例如 `作者` 或邮箱前缀
- 点开后有小菜单：`工作台` / `退出`
- 移动端：保留最小入口，避免头部拥挤

### 2.4 文案边界

推荐文案方向：

- 强调“作者登录”而非“注册/加入”
- 强调“使用已授权邮箱”而非“任选账号”
- 强调“认证 ≠ 获得权限”

避免这些表达：

- `欢迎加入`
- `注册账号`
- `创建你的账户`
- `选择你喜欢的平台`

因为当前产品本质不是开放账号系统。

### 2.5 边界与异常处理

- 同一邮箱通过不同 provider 登录，权限结论必须一致
- 若 provider 返回的邮箱为空、未验证或不稳定，要在服务端先兜底，前端只呈现清楚错误
- 登录失败页/提示应区分：认证失败、权限不足、系统错误
- 若用户已登录且有权限，再访问 `/login`，建议直接跳去 `/dashboard`
- 若用户已登录但无权限，再访问 `/login`，可停留登录页并提示当前账号无权限，允许切换 provider

### 2.6 明确 Do / Don't

**Do**

- 把登录定义为作者入口
- 三个 provider 平权呈现
- 明确解释白名单机制
- 明确区分认证成功与权限成功

**Don't**

- 不做公开注册感
- 不做复杂账号绑定/解绑 UI
- 不在 header 塞三枚 provider 按钮
- 不把无权限状态说得模糊

## 3. 多内容类型下的搜索筛选 / 结果 UX

### 3.1 目标

当前 `src/components/search-box.tsx` 已能搜文章和项目，但一旦 `posts / projects / notes / reading-list` 都接入，继续只靠单输入框 + 统一卡片，会迅速失去可读性。V2 应把搜索做成“轻筛选、强结果理解”，而不是复杂高级检索。

### 3.2 关键用户流

#### Flow A：直接搜索关键词

用户进入 `/search` 后：

1. 先看到搜索输入框
2. 输入关键词后立即出现结果
3. 顶部同步显示命中总数与当前筛选条件
4. 默认按综合相关度排序

这条流保持简单，不要被筛选器抢戏。

#### Flow B：搜索后按内容类型收缩范围

当结果混合了文章、项目、笔记、书单时，用户下一步最可能做的是“只看其中一类”。

建议在输入框下增加一行 filter chips：

- `全部`
- `文章`
- `项目`
- `笔记`
- `阅读`

交互规则：

- 默认 `全部`
- 点击即本地筛选，不重新请求
- chip 显示各类型命中数，如 `项目 3`
- 小屏允许横向滚动

#### Flow C：根据标签进一步缩小

若搜索词较宽，且结果中标签集中，建议在类型筛选下再出现“相关标签”区：

- 只展示当前结果集中出现频次最高的前 6~8 个标签
- 点选后进行二次过滤
- 多选可后置；V2 先单选足够
- 已选标签要能一键清空

这是“逐步收窄”的体验，比一开始堆一堆筛选器更轻。

### 3.3 搜索页布局建议

#### 桌面端

搜索页可升级为 `主结果区 + 右侧辅助栏`：

- 主区：输入框、filter chips、结果列表
- 右栏：
  - 当前搜索说明
  - 内容类型分布
  - 热门标签/相关标签
  - 清空筛选

但右栏只在结果较多时显示；无查询时不需要。

#### 移动端

移动端不要坚持右栏，统一下沉成：

- 输入框
- 类型 chips
- 标签 chips
- 结果列表

筛选条应 sticky，滚动时保留在顶部，方便来回切换。

### 3.4 结果卡片设计

当前搜索结果卡片已经接近可用，但在多内容类型下需要更强区分。

建议每条结果固定包含：

- `类型徽标`
  - 文章 / 项目 / 笔记 / 阅读
- `标题`
- `一句摘要`
- `元信息`
  - 日期
  - 阅读时长（适用时）
  - 项目状态（适用时）
- `标签`
- `命中提示`
  - 可选显示命中字段，如“标题匹配”/“标签匹配”/“摘要匹配”

不同类型只在元信息层有所区别，不要做四套完全不同的卡片。统一结构更利于扫描。

#### 结果视觉层级

- 类型 badge 用现有 `accent-subtle` 体系即可
- 项目状态可以复用 `ProjectCard` 的状态色
- 阅读清单结果建议显示作者/书名或来源字段（如果数据模型有）
- 笔记结果卡片应更紧凑，体现其碎片性

### 3.5 空态 / 无查询态 / 无结果态

这是搜索体验里最容易被忽略的部分。

#### 无查询态

当前只写“输入关键词开始搜索文章和项目”。V2 应改成更贴合多类型：

- 展示可搜索范围：文章、项目、笔记、阅读
- 给出 3~5 个可点的示例关键词或热门标签
- 点击示例词直接填入并触发搜索

#### 无结果态

当前无结果态已经有“去文章/项目看看”，V2 建议增强：

- 显示当前筛选条件
- 提示用户先清空类型/标签筛选再试
- 提供快速动作：`查看全部文章` / `查看全部项目` / `清空筛选`

#### 加载态

索引加载时，建议使用 3~5 条 skeleton 卡片，不只是一句“加载索引中…”。这会让页面结构更稳定。

### 3.6 交互与实现边界

V2 值得做：

- URL 同步：`q`、`type`、`tag` 都进入 query string，便于分享和返回
- 本地筛选：类型、标签都基于已加载索引本地完成
- 结果数统计：总数 + 各类型数量
- 高亮关键词：至少在标题和摘要里做轻高亮

V2 不建议做：

- 复杂多条件高级搜索面板
- 排序器过多选项
- 多标签布尔逻辑组合
- 即时跨页跳转式“全局命令面板”
- 搜索建议联想和全文高亮一起上

### 3.7 建议的组件调整

对于 `src/components/search-box.tsx`，建议拆分为：

- `SearchInput`
- `SearchTypeChips`
- `SearchTagChips`
- `SearchSummaryBar`
- `SearchResultCard`
- `SearchEmptyState`

这样后续内容类型继续增加时，不会把一个组件堆成所有逻辑入口。

### 3.8 明确 Do / Don't

**Do**

- 先给关键词搜索，再给轻筛选
- 强化类型辨识
- 强化结果解释
- 保持 URL 可分享
- 保持移动端筛选可用

**Don't**

- 不做臃肿搜索后台
- 不做四套完全不同的结果模板
- 不让筛选器比结果本身更抢眼
- 不把搜索页做成数据库查询界面

## 4. 跨三条主线的统一建议

### 4.1 统一状态语言

建议全站把状态文案统一，避免同义词混乱：

- 写作：`未保存` / `保存中` / `已保存` / `保存失败` / `存在冲突`
- 发布：`草稿` / `已发布` / `已更新`
- 认证：`未登录` / `已认证` / `无权限`
- 搜索：`全部结果` / `筛选中` / `无结果`

### 4.2 统一信息密度

当前站点的优点是克制。V2 即使增加能力，也应坚持：

- 第一屏只回答当前主任务
- 辅助说明收进次级区域
- 风险动作给明确确认
- 长解释优先拆成一行一意，不写大段帮助文案

### 4.3 对现有风格的尊重

基于 `src/app/globals.css` 和现有页面风格，以下是最稳妥的方向：

- 保留蓝白主色与浅表面层次
- 保留圆角、细边框、轻阴影
- 增加状态色，但不要把页面做成后台管理红黄绿大拼盘
- 编辑器局部允许更深色，但整体仍要像同一站点的一部分

---

## 5. 最终建议优先级

如果实现资源有限，建议 UI/交互优先顺序如下：

1. 写作工作台：自动保存 + 恢复提示 + 发布确认
2. 登录入口：统一作者入口 + 三 provider 平权 + 白名单解释
3. 搜索页：类型 chips + 多类型结果卡片 + 更完整空态
4. 写作工作台：frontmatter 分组 + tag 输入优化 + 专注模式
5. 搜索页：相关标签筛选 + 关键词高亮

这五步做完，V2 的“可用性提升”会远大于单纯多加几个字段或按钮。
