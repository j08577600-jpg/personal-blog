# personal-blog V2 实现地图

## 1. 当前系统结构：一眼看清

### 1.1 Auth 现状

当前认证是 **NextAuth + GitHub 单提供方**。

核心文件：
- `src/lib/auth.ts`
- `src/app/api/auth/[...nextauth]/route.ts`
- `src/app/login/page.tsx`
- `src/components/github-signin-button.tsx`
- `src/components/header-auth-controls.tsx`
- `src/types/next-auth.d.ts`
- `.env.example`

关键事实：
- `src/lib/auth.ts` 里只注册了 `GitHubProvider`，并依赖：`GITHUB_ID`、`GITHUB_SECRET`。
- 白名单逻辑只认 `ALLOWED_GITHUB_USERS`，并且比较的是 `token.sub`（当前语义接近 GitHub user id）。
- `requireAuthorPageSession()` / `requireAuthorApiSession()` 已经成为 dashboard / writer / API 的统一守门入口，后续扩 OAuth 时这里是第一核心点。
- 登录页与顶部登录入口都写死为 GitHub 文案与 `signIn("github")`。
- `src/types/next-auth.d.ts` 只补了 `Session.user.id/name/email/image`，**没有**补 `whitelisted`、`provider`、`providerAccountId` 等未来多提供方会需要的字段。

结论：
- 当前 auth 链路不复杂，但**耦合在“GitHub ID 白名单”这个假设上**。Google / Microsoft 接入不是简单加 provider；真正要改的是“作者身份判定模型”。

---

### 1.2 Dashboard / Writer 现状

核心页面与组件：
- `src/app/dashboard/page.tsx`
- `src/app/dashboard/write/page.tsx`
- `src/app/dashboard/write/new/page.tsx`
- `src/app/dashboard/write/[slug]/page.tsx`
- `src/app/dashboard/preview/[slug]/page.tsx`
- `src/components/writer/workbench-form.tsx`
- `src/components/writer/frontmatter-fields.tsx`
- `src/components/writer/body-editor.tsx`
- `src/components/writer/publish-toggle.tsx`
- `src/app/api/dashboard/posts/route.ts`
- `src/app/api/dashboard/posts/[slug]/route.ts`
- `src/lib/author-posts.ts`
- `src/lib/posts.ts`

当前 writer 流：
1. 页面通过 `requireAuthorPageSession()` 进入。
2. `/dashboard` 读取 `getAllPostEntries()`，展示 published / draft / invalid。
3. `/dashboard/write` 展示所有可编辑 post 条目入口。
4. `/dashboard/write/new` 用 `createEmptyDraft()` 初始化表单。
5. 表单提交到：
   - `POST /api/dashboard/posts` -> `createDraft()`
   - `PUT /api/dashboard/posts/[slug]` -> `updateDraft()`
6. 数据最终直接落到 `content/posts/*.mdx`。
7. 预览页 `/dashboard/preview/[slug]` 直接读取作者可见 post（包括 draft）。

关键事实：
- 当前 writer **只支持 posts**，完全没有 projects / reading-list / notes 的写作入口。
- `src/lib/author-posts.ts` 已经实现了：
  - slug 规范化
  - frontmatter 校验
  - 原子写入
  - mtime 冲突检测
  - 改文件名时冲突校验
  - `.bak` 备份
- 这套能力是 V2 最值得复用的基底。

结论：
- 如果 V2 要做“统一内容工作台”，最省事路线不是重写 writer，而是**把 `author-posts.ts` 的模式抽象为多内容类型 content editor service**。

### 1.3 内容模型现状

#### A. Posts
核心文件：
- `src/lib/posts.ts`
- `content/posts/*`

frontmatter schema 最完整，使用 `zod` 明确校验：
- `title`
- `date`
- `slug`
- `excerpt`
- `tags`
- `published`
- `cover?`

特征：
- 有 `invalid` 概念，dashboard 可显式暴露坏文件。
- 有 author 视角读取（`getPostBySlugForAuthor`）。
- 是唯一具备“管理台能力”的内容类型。

#### B. Projects
核心文件：
- `src/lib/projects.ts`
- `content/projects/*`
- `src/app/projects/page.tsx`
- `src/app/projects/[slug]/page.tsx`

frontmatter / data shape：
- `slug`
- `title`
- `excerpt`
- `date`
- `tags`
- `published`
- `status?`
- `featured?`
- `links?`
- `tech?`

特征：
- 没有 `zod` schema。
- `parseProject()` 缺字段时直接 `null`，没有 invalid 暴露机制。
- `getProjectBySlug()` 实际**不检查 `published`**；注释写“not published returns null”，但实现并非如此。这是一个隐性行为偏差点。
- 当前 UI 已支持状态筛选与 featured 排序。

#### C. Reading List
核心文件：
- `src/lib/reading-list.ts`
- `content/reading-list/*`
- `src/app/reading-list/page.tsx`
- `src/app/reading-list/[slug]/page.tsx`

frontmatter / data shape：
- `slug`
- `title`
- `date`
- `type`
- `status`
- `author?`
- `source?`
- `url?`
- `rating?`
- `tags`
- `published`

特征：
- 按 status 分组展示。
- 同样没有 schema 层，字段缺失直接 `null`。
- 没有 dashboard / writer / invalid 视图。

#### D. Notes
核心文件：
- `src/lib/notes.ts`
- `content/notes/*`
- `src/app/notes/page.tsx`
- `src/app/notes/[slug]/page.tsx`

frontmatter / data shape：
- `slug`
- `title`
- `date`
- `tags`
- `published`

特征：
- 最轻量。
- 也没有 schema 层与 author 管理入口。

总体判断：
- 当前 4 类内容并不是“同一内容系统”，而是 **4 套并列 loader**。
- posts 已进入“可管理内容”阶段；projects / reading-list / notes 还停在“静态读取内容目录”阶段。
- V2 如果要统一 writer / dashboard，**内容模型标准化**会比 UI 更关键。

### 1.4 Search 现状

核心文件：
- `src/lib/search.ts`
- `scripts/build-search-index.ts`
- `src/components/search-box.tsx`
- `src/app/search/page.tsx`
- `public/search-index.json`
- `package.json`

当前机制：
- build 前执行 `tsx scripts/build-search-index.ts`。
- 脚本只读取：`getPosts()` + `getProjects()`。
- 输出静态文件 `public/search-index.json`。
- 前端 `SearchBox` 用 Fuse.js 在浏览器本地检索。

当前索引范围：
- `post`
- `project`

明确缺口：
- `reading-list` 和 `notes` **没有进入搜索索引**。
- 开发期如果内容更新但没重新构建，静态索引可能过时；当前没有 dev 热更新索引机制。
- 搜索结果类型只有 `post | project`，要扩到更多内容类型必须先改 `SearchEntryType`、映射器和前端结果 UI。

结论：
- Search 不是独立系统，它直接绑在内容 loader 上。
- 任何“统一内容系统”改造，都会连带改 `search.ts` 和 `build-search-index.ts`。

## 3. 主要冲突热点（很重要）

### 热点 A：`src/lib/auth.ts`
这是 Google / Microsoft OAuth 改造第一热点。

原因：
- 现在 provider 列表只有 GitHub。
- 白名单模型只认 GitHub user id。
- `jwt` callback 里直接用 `user.id -> token.sub`，对不同 provider 的语义并不统一。

如果直接硬加：
- `GoogleProvider(...)`
- `AzureADProvider(...)` 或 Microsoft Entra ID provider

但不重构白名单模型，会立刻遇到：
- 如何区分不同 provider 的 identity？
- `ALLOWED_GITHUB_USERS` 还能不能继续用？
- 现在 dashboard 上展示的 “GitHub ID” 字段如何兼容？

建议：
- 把“白名单依据”从单一 `GitHub ID` 改成**通用 author identity**。
- 最稳的是先支持基于 email 白名单，再补 provider-specific 白名单。

---

### 热点 B：`src/types/next-auth.d.ts`
这是典型容易漏的点。

多 provider 一上来，session 往往至少需要：
- `user.id`
- `user.email`
- `user.name`
- `user.image`
- `user.whitelisted`
- `user.provider`
- `user.providerAccountId`

否则：
- 服务端能跑，TS 类型会开始漂。
- dashboard / header / login 组件改起来会不断出现 any 或类型断裂。

---

### 热点 C：`src/components/github-signin-button.tsx` + `src/app/login/page.tsx` + `src/components/header-auth-controls.tsx`
这是 UI 耦合热点。

现在问题不是按钮少，而是文案、交互、命名全部写死 GitHub：
- “GitHub 登录”
- `signIn("github")`
- Header 未登录态也是 GitHub 文案

如果 V2 目标是多 provider：
- 这些组件最好升级成 provider-agnostic 结构，至少支持按钮列表。

---

### 热点 D：`src/app/dashboard/page.tsx` / `src/app/dashboard/write/page.tsx`
这是“统一内容工作台”改造热点。

当前 dashboard 只统计 posts；writer 列表也只列 posts。

一旦要把 projects / reading-list / notes 纳入：
- 统计卡片逻辑要改
- 数据来源要抽象成 content registry / content summary
- URL 结构要重新设计（按 type 分仓，或统一 editor 路由）

这里如果不先抽象，后面会在页面层写出大量 if/else。

---

### 热点 E：`src/lib/author-posts.ts`
这是“能复用也最容易被过度复制”的热点。

危险点：
- 如果新增 `author-projects.ts` / `author-reading-list.ts` / `author-notes.ts`，直接复制这份代码，短期快，长期维护会很脏。
- 但如果一开始就抽过度通用，也容易做成大而空 abstraction。

建议：
- 先提炼**共通文件写入内核**，例如：
  - frontmatter validate
  - filename build
  - atomic write
  - mtime check
  - backup handling
- 再让每个内容类型注入自己的 schema / filename policy / defaults。

---

### 热点 F：`src/lib/search.ts` + `scripts/build-search-index.ts` + `src/components/search-box.tsx`
这是搜索扩域热点。

只要把 notes / reading-list 拉进搜索：
- `SearchEntryType` 要扩展
- `SearchEntry.section` 显示文案要扩
- `buildSearchIndex()` 参数结构要扩
- build script 要新增 loader
- 搜索 UI 空态与结果文案也应同步改为“不止文章和项目”

---

### 热点 G：内容 loader 不一致
- `posts.ts` 有 schema、invalid、author-view
- 其他 3 个 loader 都没有

这是全局结构性热点。

不统一这一层，V2 很难做真正稳定的统一后台。

---

## 4. Google / Microsoft OAuth：需要的 env 与配置变更

下面给的是**项目侧最少需要准备的配置面**，不是最终 provider 实现细节绑定某个库版本的死写法。

### 4.1 建议新增的 env

在 `.env.example` 里至少补：

```env
# Common NextAuth
NEXTAUTH_SECRET=
NEXTAUTH_URL=

# GitHub
GITHUB_ID=
GITHUB_SECRET=
ALLOWED_GITHUB_USERS=

# Google OAuth
GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=

# Microsoft OAuth / Entra ID
MICROSOFT_CLIENT_ID=
MICROSOFT_CLIENT_SECRET=
MICROSOFT_TENANT_ID=

# Recommended new allowlist model
ALLOWED_AUTHOR_EMAILS=
ALLOWED_GOOGLE_EMAILS=
ALLOWED_MICROSOFT_EMAILS=
```

更稳的收敛方案：
- 初版多 provider 建议只保留一个：`ALLOWED_AUTHOR_EMAILS`
- 如果确实要 provider 级别细分，再保留 provider-specific allowlist

### 4.2 代码侧 auth 需要改的点

#### `src/lib/auth.ts`
需要做的事：
- provider 列表扩为 GitHub + Google + Microsoft
- 在 `jwt` / `session` callback 中明确保存：
  - provider
  - providerAccountId
  - email
  - normalized whitelist result
- 把白名单逻辑从 `ALLOWED_GITHUB_USERS` 扩为可组合策略：
  - email allowlist
  - GitHub id allowlist（兼容旧逻辑）
  - provider-specific email allowlist（可选）

建议落地字段：
- `token.provider`
- `token.providerAccountId`
- `token.whitelisted`
- `session.user.provider`
- `session.user.providerAccountId`
- `session.user.whitelisted`

#### `src/types/next-auth.d.ts`
同步扩类型，避免全站 any 蔓延。

### 4.3 登录页 / 顶部入口改造

需要改：
- `src/app/login/page.tsx`
- `src/components/github-signin-button.tsx`（建议改名）
- `src/components/header-auth-controls.tsx`

建议结构：
- `AuthProviderButtons` 组件
- 按 provider 渲染按钮：GitHub / Google / Microsoft
- header 未登录态文案从 “GitHub 登录” 改成 “登录” 或 “作者登录”

### 4.4 OAuth 平台侧回调地址

项目说明需要更新到 README / 运维文档。

建议回调地址：
- GitHub: `/api/auth/callback/github`
- Google: `/api/auth/callback/google`
- Microsoft: `/api/auth/callback/azure-ad` 或该 provider 对应 id 的 callback path

注意：
- **最终 callback path 取决于你在 NextAuth 里注册的 provider id**。
- 所以 README 文档必须跟代码里的 provider id 保持一致，不能凭印象写。

### 4.5 Microsoft 特别注意

通常需要 tenant 配置。

建议 env：
- `MICROSOFT_TENANT_ID`

如果目标是仅限个人组织作者登录：
- tenant 应明确，不建议直接用 overly broad common endpoint 而不加约束。

### 4.6 白名单策略建议

推荐顺序：
1. `ALLOWED_AUTHOR_EMAILS` 作为主判定
2. 保留 `ALLOWED_GITHUB_USERS` 兼容旧逻辑
3. dashboard 显示 provider + providerAccountId，而不是继续写死 “GitHub ID”

原因：
- Google / Microsoft 的稳定主键与 GitHub id 不是同一模型。
- email 对作者权限运营更直观。
- 多 provider 共存时，email allowlist 成本最低。

## 5. 推荐开发顺序（非常建议照这个顺序）

### 阶段 1：先重构 auth 模型，不先碰 writer
目标：把“GitHub 专用认证”变成“多 provider 作者认证”。

顺序：
1. 改 `src/lib/auth.ts`，抽象 whitelist 判定
2. 改 `src/types/next-auth.d.ts`
3. 改登录页 / header 登录入口
4. 改 `.env.example` 与 README
5. 验证 dashboard 还能正常挡住未授权用户

原因：
- 后续任何作者工作台能力都依赖 auth；先稳住门禁，后面不会反复返工。

---

### 阶段 2：统一内容 schema 基础设施
目标：不要一边扩 writer，一边继续让 4 类内容各自为政。

顺序：
1. 为 `projects` / `reading-list` / `notes` 建立明确 schema（建议 zod）
2. 给 4 类内容都补齐 parse / validate / published / invalid 能力
3. 统一形成 content summary 结构，例如：
   - type
   - slug
   - title
   - date
   - published
   - status
   - validationError
   - sourceFileName

原因：
- 这是统一 dashboard / search / editor 的地基。
- 不先做这步，后面页面层会写得很丑。

---

### 阶段 3：抽 writer 的“文件操作内核”
目标：从 `author-posts.ts` 提炼共通能力。

顺序：
1. 提炼 atomic write / backup / mtime / rename-conflict 公共函数
2. 保留 posts 现有行为不变
3. 为其他类型预留 adapter 接口

原因：
- 这是最适合渐进式重构的位置。
- 先保证 posts 不回归，再扩别的内容类型。

---

### 阶段 4：扩 dashboard 为统一内容工作台
目标：dashboard 不再只看 posts。

顺序：
1. `dashboard` 加内容类型分组/筛选
2. `write` 列表页支持多类型入口
3. 决定 URL 方案：
   - 方案 A：`/dashboard/write/post/[slug]`
   - 方案 B：`/dashboard/write/[type]/[slug]`

建议：直接上 `type` 维度路由，不要继续把 post 当默认宇宙中心。

---

### 阶段 5：扩 search 到全内容域
目标：搜索真正覆盖站内内容，而不是只搜 posts / projects。

顺序：
1. 扩 `SearchEntryType`
2. 给 notes / reading-list 增 mapping
3. 改 build script
4. 改 search UI 文案与结果 badge
5. 重新生成 `public/search-index.json`

---

### 阶段 6：最后再做统一编辑器 UI 体验优化
例如：
- 每种内容类型不同的 frontmatter 表单
- autosave
- unsaved changes guard
- richer preview
- metadata helpers

原因：
- 这些都属于锦上添花，不该先于模型与权限。

---

## 6. 实施建议：最小可行 V2 范围

如果要控制范围，建议 V2 不要一次性“全统一到极致”，而是这样切：

### V2-MVP
- 多 provider auth：GitHub + Google + Microsoft
- whitelist 改为 email-first
- dashboard 先支持按内容类型概览
- search 扩到 notes + reading-list
- writer 先只继续完整支持 posts
- projects / reading-list / notes 先只做“内容可见 + 管理概览”，不急着立刻可编辑

这是最稳的。

### V2.1
- notes writer
- reading-list writer
- schema / invalid 统一

### V2.2
- project writer
- 统一 content registry
- 统一 preview model

理由：
- 当前唯一成熟写作链路是 posts。
- 一次把 4 类内容都做成全功能编辑器，冲突面会太大。

## 8. 关键风险提醒

### 风险 1：白名单迁移
如果当前线上已有 GitHub author 在用：
- 不能直接删 `ALLOWED_GITHUB_USERS`
- 需要兼容期

### 风险 2：Provider id 与 callback path 不一致
Google / Microsoft 接入最容易在 README、平台控制台、NextAuth provider id 三处写岔。

### 风险 3：未统一 schema 就扩 writer
会导致每种内容类型都手写一套 frontmatter 校验与错误处理，后面很难收口。

### 风险 4：搜索索引遗漏 unpublished 规则
当前 search build 依赖 public loaders，因此天然只索引公开内容。扩 notes / reading-list 时必须保持这个原则，别把 draft / private 内容编进静态索引。

### 风险 5：`projects` 当前行为和注释不一致
`getProjectBySlug()` 注释说“not published returns null”，但实现没有过滤 unpublished。做统一内容改造时应一起修，避免作者预览和公开访问边界混乱。

## 9. 最后结论

这个仓库现在不是“缺几个页面”，而是正处在：
- auth 从单 provider 升级到多 provider
- content 从单一 post 管理升级到多内容类型管理
- search 从局部内容升级到全站内容

最合理的推进顺序是：
1. **先 auth**
2. **再统一内容 schema**
3. **再抽 writer 内核**
4. **再扩 dashboard / writer**
5. **再扩 search**
6. **最后做编辑体验优化**

如果反过来先做 UI，会反复返工。先把身份和内容模型打平，后面就顺了。
