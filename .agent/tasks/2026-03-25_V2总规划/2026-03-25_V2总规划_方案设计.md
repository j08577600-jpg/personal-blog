# V2总规划 方案设计

> 日期：`2026-03-25`
> 任务：`V2总规划`
> 负责人：planner-codex

## 目标

为 personal-blog 的 V2 做一次性开发总规划，在不引入数据库和复杂账户系统的前提下，把现有能力整合成可落地的四条主线：

- A：写作工作台升级
- B：认证扩展
- C：内容模型统一 + 搜索增强
- D：发布运营 / 基础运维 / 最小分析反馈

本规划的核心不是简单列功能，而是先明确共享边界、主文件冲突面、前后依赖和错峰顺序，确保后续 P3/P4/P5/P6 可以按主线独立流转，但最终仍能一次性收口交付。

## 背景

现有项目已经具备 V2 的基础底盘：

- 文件驱动内容系统已形成，`posts / projects / notes / reading-list` 已不再是纯概念层。
- 在线写作工作台 V2 已上线，说明 dashboard 内作者工作流已初步打通。
- NextAuth + 作者白名单已建立最小权限边界。
- 搜索 V2 已有一轮方案，且已经意识到索引结构、内容校验链路、内容类型扩展之间存在强耦合。
- `DEVELOPMENT_PLAN.md` 已把 V2 总目标拆为写作效率、认证入口、内容模型、搜索、发布运营、数据反馈几个方向。
- `docs/V2_一次性开发编排计划.md` 已明确“一次性开发、错峰推进、统一收口”的整体节奏。

因此当前最需要的不是继续零散起子任务，而是把 V2 汇总成一份真正能指导编排的 P1/P2 总文档：

- 每条主线做什么、不做什么
- 主线之间谁先谁后、哪里能并行、哪里必须避让
- 哪些共享文件要尽量由单条主线先定口径
- 哪些风险如果不先压住，会在 P3 合流时放大成返工

## 设计原则

### 1. 文件驱动原则不变
- 内容真源仍是仓库内文件，不预埋数据库层。
- 所有统一建模都必须围绕“多目录、多 schema、统一公开消费口径”展开，而不是先造通用 CMS。

### 2. 权限判断口径不变
- 作者权限继续按 email 白名单判断。
- 新增 OAuth provider 只扩展登录入口，不改变授权语义。

### 3. 先稳共享底座，再扩功能表面
- 涉及 `src/lib/*` 的公共读取、校验、索引、auth 判定等逻辑，优先统一。
- 基于未稳定底座的 UI 或运营能力，一律后置。

### 4. 主线独立流转，但共享文件错峰修改
- 可以并行的是功能线，不是同一条共享链路的多个大改。
- 明显会改同一核心文件或同一核心数据结构的任务，不并行开发。

### 5. 一次性交付不等于一次性大爆炸
- 采用分波次推进：A/B 先行，C 后置整合，D 在 C 稳定后落地，最后统一收口。
- 每条主线都要先定义本线验收口径，再进入实现。

## 主线拆分

---

## A：写作工作台升级

### 目标

把当前“能写、能存、能发”的在线工作台，提升为作者高频可用、对丢稿和误操作更有韧性的工作面。

### 范围

- 自动保存机制与明确的保存状态反馈
- 刷新/异常退出后的编辑恢复策略
- 发布确认、取消发布确认、防误操作优化
- frontmatter 编辑体验升级：`slug / excerpt / tags / cover / date / published`
- 写作页信息层级、错误提示、字段顺序优化
- 最近一次备份或最近差异的可读化展示（轻量方案）
- 与预览、保存、发布状态联动的反馈一致性

### 边界

- 不引入多人协作、评论、审批流、版本历史时间线
- 不引入富文本编辑器、拖拽式 block editor、媒体库
- 不扩展到 `projects / notes / reading-list` 的统一后台编辑器
- 不在本阶段重构底层内容 schema，只围绕 `posts` 的作者态体验增强
- 不承诺完整 Git 式 diff 或任意历史恢复

### 依赖

- 依赖当前在线写作工作台 V2 已有页面骨架和受保护写入链路
- 依赖现有作者鉴权可稳定识别登录作者
- 依赖 `posts` 内容 frontmatter schema 在本线内保持稳定，避免与 C 同时重构
- 与 C 在 `dashboard` 内容读取链路、内容序列化、公共类型层存在潜在交集，必须错峰

### 主要风险

- 自动保存与恢复会改动写入频率和本地状态管理，若同时叠加内容模型改造，极易引发回归
- 写作工作台若直接提前适配“多内容类型统一编辑”，范围会迅速失控
- 最近差异/备份展示若实现过重，会把本线拖向版本系统
- 若状态反馈设计不清晰，新增功能会放大作者操作焦虑，而不是降低焦虑

### 风险应对

- A 线只服务 `posts`，不提前背 C 线统一编辑负担
- 差异展示限定为“最近一次备份/最近一次保存前后摘要”，不做全历史
- 自动保存采用明确节流策略与可见状态提示，不做过度复杂的离线同步
- 与 C 错峰：A 先稳定作者端写作主链路，C 再碰公共内容模型

### 验收标准

- 作者在写作页持续输入时，可看到稳定、可理解的保存状态变化
- 刷新、意外关闭或保存失败后，至少有一条明确恢复路径，不再只依赖手工复制
- 发布与取消发布都有明确确认或足够强的防误触反馈
- 常见 frontmatter 错误可在字段附近直接理解并修正
- `posts` 的新建、编辑、保存、预览、发布主链路不因增强而退化

### 顺序建议

- 第一波优先实现，并在 C 开始前完成 P3/P4 主体
- 原因：A 会碰 dashboard 写入链路、作者态表单和内容文件落盘，是后续内容统一前必须先稳住的高频主链路
- 与 B 可并行；与 C 不建议同时进入核心改造期

---

## B：认证扩展（Google + Microsoft OAuth）

### 目标

在保持 email 白名单授权语义不变的前提下，把登录入口从单 provider 扩展到 Google 和 Microsoft，提高作者登录可用性与冗余度。

### 范围

- 接入 Google OAuth provider
- 接入 Microsoft OAuth provider
- 登录页文案、按钮顺序、错误提示、provider 展示统一
- 统一 session 中的 email 判定口径
- 验证同邮箱不同 provider 登录时的作者权限行为一致
- 明确环境变量配置和缺省行为

### 边界

- 不做账号中心
- 不做用户资料编辑
- 不做复杂账号合并 UI
- 不引入数据库型用户表或身份绑定表
- 不改变作者权限规则：仍按 email 白名单授权
- 不引入团队成员管理后台

### 依赖

- 依赖现有 NextAuth 体系与白名单逻辑
- 依赖 provider 返回 email 且满足项目授权口径
- 依赖登录页/受保护页面已有基础路由
- 对 C/D 依赖低，主要是对站点全局 session 行为有横向影响

### 主要风险

- 不同 provider 的 email claim 大小写、是否 verified、字段名来源不同，可能导致白名单口径不一致
- 现有白名单若仍残留 provider-specific 判断，新增 provider 后可能出现“登录成功但权限错判”
- 多 provider 同时展示后，登录页文案与错误态可能混乱
- 本地/线上环境变量遗漏会造成部分 provider 在特定环境不可用

### 风险应对

- 在 auth 层收口成统一的 email normalization 和 author check
- 把 provider 差异限制在登录与 profile 映射层，不渗透到 dashboard 授权层
- 明确“OAuth 成功 != 获得作者权限”的交互提示
- 环境变量缺失时，provider 应有可见降级策略，而不是静默坏掉

### 验收标准

- Google 登录成功后，白名单邮箱可进入作者后台，非白名单不可进入作者能力
- Microsoft 登录成功后，白名单邮箱可进入作者后台，非白名单不可进入作者能力
- 同一白名单邮箱通过不同 provider 登录，权限结论一致
- 登录页清楚展示可用 provider 和失败提示，不制造歧义
- 现有登录、退出、受保护页面访问链路无回归

### 顺序建议

- 第一波与 A 并行实施
- 原因：B 对共享内容文件改动极少，主要集中在 `auth`、登录页和 session 映射层，是低冲突高收益任务
- 应优先于 D，且可在 C 进入实现前完成 P4/P5

---

## C：内容模型统一 + 搜索增强

### 目标

把 `posts / projects / notes / reading-list` 从“并列存在但口径分散”的状态，收束成统一的公开内容模型与搜索索引模型，为站点继续增内容类型提供低维护成本的底盘。

### 范围

- 梳理四类内容的公共字段、差异字段、对外可见性语义
- 统一公开索引字段：至少包括类型、标题、摘要、标签、日期、链接、搜索附加文本等
- 统一列表展示和详情页读取时共享的最小结构
- 统一搜索索引生成链路，确保只索引“对外可见且结构有效”的内容
- 搜索接入更多内容类型，并引入类型/标签等轻量筛选
- 优化中文短 query 的召回与排序策略，控制在轻量级范围
- 清理现有重复解析、绕过 schema 校验、展示口径不一致的问题

### 边界

- 不引入数据库或外部搜索服务
- 不做完整 CMS 化内容注册系统
- 不做一次性支持无限内容类型的插件框架
- 不做重型中文分词、拼音、同义词后台
- 不强行把作者工作台统一成四种内容共用编辑器
- 不在本阶段先做运营能力 UI，避免建立在未稳定模型上

### 依赖

- 依赖 A 先稳定 `posts` 作者侧链路，避免同时改写入与读取主路径
- 依赖现有 `projects / notes / reading-list` 内容目录和读取能力作为输入样本
- 与 D 高耦合：推荐位、系列、置顶、更新记录、分析指标都依赖统一内容标识和展示字段
- 会涉及 `src/lib/*`、搜索构建脚本、列表页/搜索页共享组件等公共区域，必须拿到单线改造窗口

### 主要风险

- 这是 V2 冲突面最大的主线，若与 A/D 同时推进，极易在公共类型定义、内容读取、搜索索引脚本上互相踩踏
- 若过早追求泛型和高度抽象，容易偏离“文件驱动 + 低复杂度”的项目原则
- notes / reading-list 与 posts / projects 展示差异大，若强行完全同构，可能牺牲实际体验
- 搜索增强若与内容统一一起无限扩容，会把范围膨胀成独立大工程

### 风险应对

- 明确分两层统一：先统一公开消费模型，再保留各类型差异字段在各自 lib 内处理
- 搜索 V2 只落地轻量优化：统一索引、轻筛选、轻排序，不做大型搜索基础设施改造
- 先统一数据契约，再改搜索页 UI；避免“先做界面后补模型”
- notes / reading-list 允许映射到统一最小结构，但不强求与文章完全同样展示

### 验收标准

- 至少四类内容都能映射到统一公开内容结构，且字段语义明确
- 搜索索引只包含结构有效、允许公开展示的内容
- 搜索结果可以稳定区分内容类型，并支持最小可用筛选
- 新增内容类型时，不再需要从零定义一套搜索/列表/展示口径
- 现有公开内容页、列表页和搜索页在接入统一模型后无明显回归

### 顺序建议

- 第二波实施，晚于 A/B，早于 D 的实现落地
- 原因：C 需要单独的公共层改造窗口；若 A 尚未稳定，C 会与作者端写入模型互相干扰；若 D 先落地，则 D 很可能建立在旧内容口径上，后续返工

## D：发布运营 / 运维 / 分析基础

### 目标

在内容与搜索底盘稳定后，补足站点从“能发布”走向“可持续运营”的最小能力，包括发布约束、首页编排、更新记录、运行检查和轻量分析反馈。

### 范围

- 发布前 checklist
- 推荐位 / 置顶机制 / 系列组织能力
- 更新记录或内容维护动作支持
- 发布/运营相关的最小后台规则或前台呈现
- 生产发布与运行检查的最小规范化
- 明确最小分析指标：访问、来源、热门内容、搜索关键词或等价轻量指标
- 选择轻量实现方案，避免重基础设施

### 边界

- 不做完整运营后台
- 不做复杂排期系统，除非真实需求明确且已被本阶段收敛
- 不做重型 BI、事件平台、数据仓库
- 不在 C 未稳定前绑定新的内容组织字段
- 不做与内容模型解耦失败的硬编码推荐系统

### 依赖

- 强依赖 C：推荐位、系列、置顶、更新记录都需要统一内容标识和展示字段
- 弱依赖 A：发布 checklist 与作者操作流程最好复用写作工作台中已有状态
- 弱依赖 B：若后台存在受保护运营入口，需要沿用认证扩展后的授权体系
- 依赖现有部署/监控基础文档与已存在的监控日志轮转经验

### 主要风险

- 若 D 在 C 之前落地，会把推荐位、系列、置顶等逻辑写死在旧模型上，形成二次返工
- 运营需求天然容易发散，若没有最小可用边界，会从“推荐位”蔓延到“完整 CMS 面板”
- 分析能力若直接走外部重型服务，复杂度会远超本项目当前阶段收益
- 发布前 checklist 若只停留在文档层，不嵌进实际发布流程，最终会流于形式

### 风险应对

- D 只做“建立基本秩序”，不做完整运营系统
- 内容组织能力优先采用文件 frontmatter / 统一内容元数据可表达的方案
- 分析能力先选轻量脚本或轻量服务，满足判断所需即可
- 将 checklist 尽量前移到作者/发布动作附近，而不是只放在 docs 里

### 验收标准

- 发布前至少有一套可执行的 checklist，并能覆盖主要失误场景
- 首页或内容入口具备最小主动编排能力，不再完全被动按时间堆列表
- 系列/置顶/推荐中的至少一组能力可实际使用
- 最小分析指标可被获取并支持内容判断，不再完全凭感觉
- 相关能力建立在统一内容模型上，而不是内容类型特判堆积

### 顺序建议

- 第三波实施，必须在 C 进入稳定状态后进行
- 原因：D 高度依赖统一内容模型，过早落地会把旧结构固化为新的运营耦合点

## 技术选择

### 方案 A：按四条主线独立立项，但分波次错峰推进（选定）

做法：

- A/B 先行并行
- C 在 A/B 基本稳定后单独推进公共层改造
- D 在 C 稳定后落地
- 最后统一 P6 收口

优点：

- 与现有 `docs/V2_一次性开发编排计划.md` 一致
- 能在不失去“一次性交付”目标的前提下控制共享文件冲突
- 每条主线都有清晰 P1/P2/P3/P4/P5/P6 流转边界

缺点：

- 需要更强的编排纪律，不能看见空档就随手插任务
- 总计划文档需要比单功能计划更明确冲突边界

### 方案 B：四条主线全部并行开发

优点：
- 表面上最快

缺点：
- `auth`、`dashboard`、`src/lib`、搜索索引脚本、公共类型层会同时被多线修改
- 合流和回归压力会在 P4/P5 集中爆炸
- 与“明显改同一核心文件不并行开发”的协作规范冲突

结论：不选。

### 方案 C：先统一内容模型，再做 A/B/D

优点：
- 公共底座最先收束

缺点：
- A 的作者高频问题继续悬而未决
- B 的认证扩展本可低成本并行，却被无谓阻塞
- C 是最重、最易发散的任务，不适合作为 V2 第一落点

结论：不选。

## 共享文件冲突面判断

高冲突区域：

- `src/lib/auth.ts` 及 NextAuth 配置链路
- `src/app/login/*`、`src/app/dashboard/*`
- `src/lib/posts.ts` 以及后续统一内容读取/索引工具
- `scripts/build-search-index.*`
- 搜索页与共享结果组件
- 与内容展示/列表相关的共享类型定义

冲突控制原则：

- A 优先占用 dashboard 写作链路
- B 优先占用 auth/session/login 链路
- C 单独占用内容读取/索引/公共模型链路
- D 在 C 后接入内容组织和分析字段，不抢公共模型定义权

## 风险与取舍

- V2 最大风险不是单个功能难做，而是多个功能同时争抢同一批公共文件。
- 若不先写清共享边界，builder 很容易在 P3 阶段顺手“顺便一起改”，最终把 A/C/D 混成一个大任务。
- 本规划刻意把 D 后置，是主动接受“运营能力晚一点落地”，换取底盘稳定。
- 本规划刻意不让 A 提前承担“多内容类型统一编辑器”，是主动接受“作者体验先只优化 posts”，换取高频链路先做对。
- 本规划刻意不把 C 做成未来无限扩展架构，是为了守住当前项目“文件驱动、低复杂度、真实需求优先”的边界。

## 结论

V2 采用“四主线、三波次、统一收口”的一次性开发方案：

- 第一波：A 写作工作台升级、B 认证扩展并行
- 第二波：C 内容模型统一 + 搜索增强单独推进
- 第三波：D 发布运营 / 运维 / 分析基础落地
- 最后统一进入 P6 合并、发布、健康检查和最终报告

该方案兼顾了三件事：

- 保证 V2 关键用户价值先落地
- 尽量减少共享文件上的开发冲突
- 为后续 builder / reviewer / tester 提供明确的任务切面和回流边界
