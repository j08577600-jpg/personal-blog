## 三、P4 / P5 回流记录与最终放行口径

### 第一次回流：P4 → P3

- 回流原因：审查阶段发现阻塞项，主要集中在公开读取边界与编辑交互语义。
- 关键问题：
  - 公开页可能暴露草稿
  - 已发布文章的作者预览链路异常
  - 发布状态卡片与主按钮存在双重切换歧义
  - slug 唯一性校验不足，存在不同文件共享同一 slug 风险
- 回修结果：
  - 公开 `getPostBySlug()` 仅返回已发布文章
  - 作者预览改走作者态读取 `getPostBySlugForAuthor()`
  - `PublishToggle` 改为只读展示，发布动作统一走主提交流程
  - 新增 `ensureSlugNotUsedByOtherFile()`，覆盖 create / update 两条链路
- 回流后结论：P4 阻塞项关闭，可重新进入测试。

### 第二次回流：P5 → P3

- 回流原因：测试阶段发现服务端真实运行链路仍有阻塞级缺陷。
- 关键问题：
  - 新建草稿仅靠前端限制，服务端未兜底强制 `published=false`
  - `cover` 缺省时，序列化会把 `cover: undefined` 传给 `gray-matter.stringify()`，触发 `YAMLException`
- 回修结果：
  - `createDraft()` 在服务端统一强制 `published: false`
  - `serializePostFile()` 在序列化前通过解构彻底移除 `cover` 字段，避免 `undefined` 落入 YAML
  - 以 `npx tsx` 真实直调 `createDraft()` / `updateDraft()` / `getPostBySlug()` 完成闭环复核，而非仅停留在静态代码意图
- 回流后结论：P5 阻塞项关闭，重新复测通过。
