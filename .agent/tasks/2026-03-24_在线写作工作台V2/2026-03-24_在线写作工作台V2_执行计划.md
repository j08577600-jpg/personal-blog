本执行计划是交给 `builder-codex` 的自包含实施文档。目标是基于现有 dashboard / preview / file-driven MDX 能力，补齐最小可交付的在线写作闭环，不写数据库，不扩范围到协作系统。

## 目标

本功能完成后，授权作者可以在站内完成以下闭环：

- 新建草稿
- 编辑 frontmatter
- 编辑正文
- 保存草稿
- 保存后预览
- 发布
- 取消发布

同时系统保持以下特性：

- 内容真源仍是 `content/posts/*.mdx`
- 公开博客继续只读已发布文章
- dashboard 写作能力仅限登录且白名单授权用户
- 保存失败、文件冲突、文件名变化都有明确错误提示和恢复方式

## 决策日志

- Decision：在线写作工作台仍以 `content/posts/*.mdx` 为唯一真源，不引入数据库。
  原因：项目现有文档明确要求 V1/V2 先文件驱动，当前闭环的核心缺口是“站内编辑能力”，不是存储层升级。
  日期/作者：2026-03-24 / planner-codex

- Decision：预览采用“保存后预览”，不做未保存内容实时预览。
  原因：现有 `/dashboard/preview/[slug]` 已可复用；若支持未保存预览，需要引入临时态传输或缓存，范围明显扩大。
  日期/作者：2026-03-24 / planner-codex

- Decision：并发冲突采用 `mtimeMs` 乐观检测，不做编辑锁或自动 merge。
  原因：最小成本避免静默覆盖，足够支撑单作者/低并发场景。
  日期/作者：2026-03-24 / planner-codex

- Decision：保存采用“完整序列化覆盖 + 临时文件 + rename”，不做正文局部 patch。
  原因：文件驱动架构下更容易保证写入一致性与恢复语义。
  日期/作者：2026-03-24 / planner-codex

- Decision：tags 输入先用简单文本框，不做复杂标签组件。
  原因：降低 UI 和状态管理复杂度，先保最小闭环。
  日期/作者：2026-03-24 / planner-codex

- Decision：发布 / 取消发布不单独拆 `PATCH`，统一复用 `PUT` 保存链路。
  原因：减少协议面，保证发布动作不会旁路校验、冲突检测与 `.bak` 备份语义。
  日期/作者：2026-03-24 / builder-codex

- Decision：公开页 slug 读取与作者态预览读取彻底分层；公开页只返回 `published` 文章，作者侧预览单独允许读取草稿与已发布文章。
  原因：修复公开路径暴露草稿，同时保留作者在编辑链路中的稳定预览能力。
  日期/作者：2026-03-24 / builder-codex

- Decision：发布状态卡片不再承担切换动作，只展示当前状态；发布/取消发布只保留主按钮这一条提交路径。
  原因：消除本地状态切换与提交动作双重反转造成的误发布风险。
  日期/作者：2026-03-24 / builder-codex

- Decision：新建页强制以草稿语义创建，创建完成前禁用发布按钮。
  原因：避免新建流程因为默认值或交互误触直接生成已发布文章。
  日期/作者：2026-03-24 / builder-codex

- Decision：`updateDraft` 在文件名变更前先做“slug 是否已被其他文件占用”校验，而不是只检查目标文件路径。
  原因：同一 slug 可以映射到不同日期文件名，单纯检查路径不足以阻止公开路由歧义。
  日期/作者：2026-03-24 / builder-codex

- Decision：`createDraft` 在服务端统一强制 `published=false`，不接受 API/函数层传入的已发布初始态。
  原因：P5 已验证仅靠前端 `forceDraftOnCreate` 仍可被绕过，必须把“新建即草稿”下沉到服务端创建链路。
  日期/作者：2026-03-25 / builder-codex

- Decision：序列化 frontmatter 时必须先从对象结构上移除 `cover` 字段，再把结果交给 `gray-matter.stringify`；不能只靠对象展开追加同名键覆盖。
  原因：真实复测证明 `...frontmatter` 仍会把 `cover: undefined` 留在对象里，`gray-matter` 依旧会抛 `YAMLException`；必须按真实运行结果彻底删掉该键。
  日期/作者：2026-03-25 / builder-codex

- 如果在 Step 1 中断：先确认是否已有统一 author guard 和 API 契约草稿，再继续后续实现，避免权限判断分叉。

## 恢复方式

### 文件写入失败恢复
- 临时文件写入或 rename 失败时，不覆盖原文件。
- 页面保留当前表单状态，用户可重试或手动复制内容。

### 并发冲突恢复
- API 返回 `409` 时，前端提示用户刷新最新版本。
- 用户可先复制当前输入，再重新加载页面手动合并。

### 文件名变更中断恢复
- 若新文件已成功写入但旧文件删除失败：
  - 视为新文件已生效；
  - 页面提示旧文件待人工清理；
  - 不回滚新文件。

### 误发布/误取消发布恢复
- 通过立即反向切换 `published` 恢复可见性。
- 若涉及正文误改，使用最近一次 `.bak` 手工恢复；本轮不承诺多版本回退。
