# 在线写作工作台V2 方案设计

> 日期：`2026-03-24`
> 任务：在线写作工作台V2
> 负责人：planner-codex

## 目标

在现有 `dashboard` + `preview` + 文件驱动 MDX 基础上，补齐一个最小可交付的作者闭环：授权作者可以在站内新建草稿、编辑 frontmatter、编辑正文、保存、预览、发布/取消发布，而公开站点仍继续只消费 `content/posts/*.mdx`。

## 背景

当前项目已经具备三块基础能力：

- `src/app/dashboard/page.tsx` 已可展示已发布 / 草稿 / 待修复三类文章状态。
- `src/app/dashboard/preview/[slug]/page.tsx` 已可预览草稿 MDX 渲染结果。
- `src/lib/posts.ts` 已以内置 Zod 校验 frontmatter，并以文件系统作为唯一内容源。

但现状仍是“线下写文件、线上只查看”模式：

- 新建草稿依赖手动在 `content/posts/` 下创建文件。
- frontmatter 与正文依赖外部编辑器修改。
- 发布 / 取消发布依赖手动改 `published` 字段。
- 保存失败、文件冲突、误发布后的恢复路径没有站内闭环。

因此需要一个 V2 最小闭环，把“作者操作”收回到受保护的 dashboard 内，同时不破坏 V1 文件驱动架构。

## 现状结论

基于当前代码与文档，可得出以下约束：

- 内容真源仍是 `content/posts/*.mdx`，公开博客页继续走 `getPosts()` / `getPostBySlug()`。
- dashboard 权限边界当前依赖 NextAuth + GitHub 白名单，已有基础守卫。
- preview 路由已存在，说明“作者侧路由”和“公开路由”已经分层。
- 当前没有任何站内写文件 API，也没有草稿锁、版本库、数据库。
- `src/lib/posts.ts` 中 `getPostBySlug()` 当前会返回匹配 slug 的文章对象，实际可读到 draft；因此 V2 设计必须继续坚持“公开页面控制展示，作者页面控制编辑”，不要把写入能力直接暴露给公共路由。

## 范围

### 做
- 在受保护的 dashboard 区域新增“在线写作工作台 V2”入口与页面骨架。
- 支持新建文章草稿：自动生成合法初始 frontmatter 与文件名。
- 支持编辑 frontmatter：至少覆盖 `title`、`date`、`slug`、`excerpt`、`tags`、`cover`、`published`。
- 支持编辑正文：以原始 Markdown/MDX 文本为单位编辑，不做富文本。
- 支持保存草稿：将表单内容序列化为完整 MDX 文件并写入 `content/posts/`。
- 支持预览：复用现有 `/dashboard/preview/[slug]` 或其渲染能力，保证预览与正式页一致。
- 支持发布 / 取消发布：本质是受控修改 `published` 字段，并保留保存前校验。
- 明确写文件策略、冲突策略、错误处理、恢复策略。
- 为 builder 提供清晰文件清单、依赖、测试点、验收标准。

### 不做
- 不引入数据库、ORM、对象存储、GitHub API 写仓库、Headless CMS。
- 不做多人协作、评论、批注、审批流、PR 审核流。
- 不做自动保存、离线草稿箱、版本 diff、内容历史时间线。
- 不做图片上传、媒体库、封面裁剪、拖拽组件化编辑器。
- 不做跨内容类型统一编辑（`notes` / `projects` / `reading-list` 先不纳入）。
- 不做公开分享预览链接，不做 token 化临时预览。
- 不做复杂 SEO 字段扩展，不新增 `scheduledAt`、`canonical`、`author` 等未来字段。
- 不做 slug 全局唯一索引系统，只做 `content/posts/` 范围内冲突检查。

## 核心方案

### 1. 路由结构

采用独立作者工作台路由，不污染公开博客路由：

- `src/app/dashboard/write/page.tsx`
  - 工作台首页，展示新建入口与草稿列表跳转。
- `src/app/dashboard/write/new/page.tsx`
  - 新建草稿页。
- `src/app/dashboard/write/[slug]/page.tsx`
  - 编辑已有草稿或已发布文章。
- `src/app/api/dashboard/posts/route.ts`
  - `POST`：创建新草稿。
- `src/app/api/dashboard/posts/[slug]/route.ts`
  - `GET`：读取作者编辑态原始内容。
  - `PUT`：保存编辑结果。
  - `PATCH`：只切换发布状态时可选使用；若 builder 认为维护成本更高，可统一走 `PUT`。

说明：

- 页面路由只做鉴权、展示和表单提交流程。
- 文件读写只允许在受保护 API Route 内执行。
- 公开 `/blog/*`、`/dashboard/preview/*` 不直接承担写入职责。

### 2. 权限边界

所有写入相关能力必须满足以下边界：

- 未登录：重定向 `/login` 或 API 返回 `401`。
- 已登录但不在白名单：页面重定向 `/unauthorized`，API 返回 `403`。
- 仅 `dashboard/write*` 页面与 `api/dashboard/posts*` API 可以读写草稿源文件。
- 公开博客页面只读已发布内容，不能承接任何写入参数。
- preview 页面继续只读，不接收任意正文直传，避免未保存内容绕过文件校验链路。

建议 builder 在 `src/lib/auth.ts` 现有 session 能力基础上，抽一个 server-side author guard（例如 `requireAuthorSession`）统一给 dashboard 页面和 API 复用，避免权限判断散落。

### 3. 文件写入策略

V2 继续坚持“文件系统是真源”，但写入过程必须可预测、可恢复。

#### 3.1 新建草稿

新建时由服务端生成：

- 默认文件名：`yyyy-MM-dd_<slug>.mdx`
- 初始 slug 生成规则：
  - 优先来自用户输入标题做 slugify；
  - 若标题为空，先用 `untitled-<timestamp>` 兜底；
  - slug 只允许小写字母、数字、中横线。
- 初始 frontmatter：

```yaml
---
title: ""
date: "yyyy-MM-dd"
slug: "generated-slug"
excerpt: ""
tags: []
published: false
---
```

正文初始值可为空，或放一行提示文本；以空正文优先，避免污染内容。

#### 3.2 保存策略

每次保存按“完整覆盖单文件”处理，不做局部 patch：

1. 页面提交结构化 frontmatter + body。
2. 服务端先做字段级校验。
3. 校验通过后统一序列化为完整 MDX 文本。
4. 先写入临时文件，再原子替换目标文件。

推荐写法：

- 目标文件：`content/posts/yyyy-MM-dd_slug.mdx`
- 临时文件：同目录下 `.<filename>.tmp`
- 备份文件：`.agent/tasks/...` 不承接运行时备份；运行时备份建议放在 `content/.trash/` 或 `content/posts/.history/` 下，按项目实际取其一。

本次建议采用更小实现：

- 临时文件 + 原子 rename 为必做。
- 运行时备份只在“文件名变化”或“发布状态切换”前保留最近一次 `.bak` 为必做。
- 多版本历史为不做。

#### 3.3 文件名变更策略

当用户修改 `date` 或 `slug` 时，目标文件名会变化。

处理原则：

- 旧文件不直接覆盖为新路径，必须走“新文件写入成功后，再删除旧文件”的双阶段流程。
- 若新路径已存在且不是当前文章自身，保存失败，提示 slug/date 冲突。
- 若新文件写入成功但旧文件删除失败，系统提示“内容已保存到新文件，但旧文件待清理”，不回滚新文件。

### 4. 数据模型与编辑形态

编辑页的最小表单模型：

- `title: string`
- `date: string`
- `slug: string`
- `excerpt: string`
- `tagsText: string`（UI 用逗号分隔输入，服务端再转 `string[]`）
- `cover?: string`
- `published: boolean`
- `body: string`
- `sourceFileName: string`
- `lastKnownMtimeMs: number`

说明：

- `lastKnownMtimeMs` 用于最小并发冲突检测。
- tags 在 UI 层不做复杂 token editor，先用简单文本输入降低范围。
- 正文编辑器先用 `textarea` 或轻量 code editor，不要求富文本、快捷命令、拖拽块。

### 5. 并发与冲突策略

V2 不做多人协作锁，但必须避免静默覆盖。

采用“最后修改时间戳”乐观并发：

- 编辑页加载时读取文件 `mtimeMs`。
- 保存时提交 `lastKnownMtimeMs`。
- 服务端保存前重新读取文件实际 `mtimeMs`。
- 若不一致，返回 `409 Conflict`，阻止覆盖。

前端收到 `409` 后展示三选一文案策略：

- 刷新并加载最新版本。
- 复制当前未保存内容后手动合并。
- 放弃本次保存。

本次不做自动 merge。

### 6. 校验策略

保存前必须做两层校验：

#### 6.1 表单层基础校验
- `title` 非空
- `date` 符合 `yyyy-MM-dd`
- `slug` 非空且匹配 slug 规则
- `excerpt` 非空
- `tags` 可为空数组
- `body` 可为空

#### 6.2 文件层约束校验
- `slug` 与目标文件名中的 slug 一致
- `date` 与目标文件名中的 date 一致
- `content/posts/` 内不存在其他文件占用相同 `date + slug`
- 序列化后的 frontmatter 再经过与 `src/lib/posts.ts` 一致的 schema 验证

建议 builder 在 `src/lib/posts.ts` 基础上补一个“作者态可写模型”的工具层，而不是在 API Route 里手搓校验逻辑。

### 7. 预览策略

预览沿用“保存后预览”，不做未保存内容实时预览。

原因：

- 现有 `/dashboard/preview/[slug]` 已能复用正式渲染能力。
- 若支持“未保存内容实时预览”，需要额外临时态传输或服务端缓存，会明显扩大范围。
- 保存后预览足以形成最小闭环：编辑 -> 保存 -> 打开预览。

因此 V2 明确规定：

- 点击“预览”前必须先保存成功。
- 新建草稿在首次创建成功后才能进入编辑态与预览态。
- 编辑页内可提供“保存并预览”按钮，成功后跳转 `/dashboard/preview/[slug]`。

### 8. 发布 / 取消发布策略

发布本质上仍是 `published` 布尔值切换，但要受控：

- 发布前必须通过完整校验。
- 发布成功后，文章应立即在 dashboard 标记为“已发布”，并可由公开 `/blog` 消费。
- 取消发布后，文章应从公开 `/blog` 消失，但仍保留在 dashboard 中可编辑、可预览。

按钮语义：

- 当前 `published: false` → `发布`
- 当前 `published: true` → `取消发布`

V2 不做二次确认弹窗强依赖；但建议对“取消发布”加轻量确认文案，防止误触。

### 9. 错误处理

按错误类型明确返回：

| 场景 | 页面/API 行为 | 用户提示 |
|------|---------------|----------|
| 未登录 | redirect `/login` / `401` | 请先登录 |
| 未授权 | redirect `/unauthorized` / `403` | 你没有写作权限 |
| frontmatter 校验失败 | `400` | 标注具体字段错误 |
| slug/date 冲突 | `409` | 文件名冲突，请修改 slug 或日期 |
| 文件被他人改动 | `409` | 文件已更新，请刷新后重试 |
| 文件不存在 | `404` | 草稿不存在或已被移动 |
| 读写异常 | `500` | 保存失败，请重试；原文件未覆盖 |

错误展示原则：

- 字段错误贴近字段显示。
- 系统错误显示全局 toast 或页面告警条。
- 不向用户暴露完整服务器堆栈。

### 10. 回滚与恢复策略

V2 至少要覆盖三类恢复：

#### 10.1 单次保存失败
- 临时文件写入失败或 rename 失败时，保留原文件不变。
- 前端继续保留编辑器内存态，让作者可复制内容后再试。

#### 10.2 文件名变更中断
- 新文件写成功、旧文件删除失败：提示新路径已生效，旧文件需人工清理。
- builder 需在计划中加入这一验收点。

#### 10.3 误发布 / 误取消发布
- 不做版本系统。
- 使用“立即反向操作”恢复：发布错了就取消发布；取消发布错了就重新发布。
- 若正文误改且已保存，只能依赖最近一次 `.bak` 进行手工恢复。

因此本方案对“恢复”做出的最低承诺是：

- 原子写入避免损坏当前文件。
- 最近一次备份避免文件名变化或状态切换时完全不可回退。
- 不承诺任意时间点历史版本恢复。

## 技术选择

### 方案 A：直接在 page Server Action 中写文件

优点：
- 文件少，接线快。

缺点：
- 权限、校验、读写、序列化耦合在页面内，后续测试与复用差。
- 新建 / 编辑 / 发布接口边界不清楚。
- 失败恢复和并发处理更难做干净。

结论：不选。

### 方案 B：受保护页面 + 受保护 API Route + 文件工具层（选定）

优点：
- 与当前 App Router 架构兼容。
- 写入边界清楚，便于权限审计。
- 更容易做保存、冲突、回滚、错误码管理。
- 不需要引入数据库，保持 V1/V2 连续性。

缺点：
- 文件数会增加。
- 需要设计前后端数据契约。

结论：选定。

### 方案 C：直接引入数据库草稿表

优点：
- 并发、版本、结构化查询都更好扩展。

缺点：
- 明显违背当前“先文件驱动，后数据库增强”的项目约束。
- 对本任务最小闭环来说是过度设计。

结论：不选。

## 风险与取舍

- 文件系统不是事务型数据库；本方案用“临时文件 + rename + 最近一次备份”做最小兜底，不追求完整历史版本。
- 当前 `getPostBySlug()` 与作者态读取模型并未完全分层；V2 实施时若继续让 API 和公开页共享同一读取函数，容易埋权限语义混乱。建议 builder 明确区分“公开只读”和“作者可读写”工具。
- 只用 `mtimeMs` 做冲突检测，能防静默覆盖，但不能解决多人合并；这是本次接受的取舍。
- 保存后预览而非实时预览，牺牲一点体验，换取范围稳定。
- tags 先用逗号文本输入，不做复杂标签组件，是为了降低实现与测试成本。
- 若需要更顺手的 UI 文案或交互稿，可作为前置条件让 `designer-gemini` 提供简版表单文案与按钮状态文案，但不阻塞最小闭环开发。

## 前置条件

非阻塞前置条件：

- 可补一份极简 UI 文案 / 交互稿，明确：按钮命名、保存中/失败提示、冲突提示、取消发布确认文案。
- 需确认运行环境对 `content/posts/` 目录有写权限；若生产环境只读，本功能只能先作为本地/受控部署功能启用。

## 测试点

- 新建草稿成功，自动生成合法文件名与默认 frontmatter。
- 新建时 slug 冲突被阻止，原目录无脏文件残留。
- 编辑 frontmatter 后保存成功，文件内容序列化正确。
- 修改 `date` / `slug` 导致文件名变更时，新文件生效，旧文件按策略处理。
- 草稿保存后能从 `/dashboard/preview/[slug]` 正常预览。
- 发布后文章出现在 `/blog`；取消发布后从 `/blog` 消失但 dashboard 仍可见。
- 并发冲突返回 `409`，不会静默覆盖。
- 保存失败时原文件不损坏，前端编辑内容仍在。
- 未登录 / 未授权无法访问写作页面和写作 API。
- `npm run lint`、`npm run build` 通过；如补测试则对应单测/e2e 通过。

## 结论

在线写作工作台 V2 采用“受保护 dashboard 页面 + 受保护 API Route + 文件工具层”的最小闭环方案，继续以 `content/posts/*.mdx` 为唯一真源，不引入数据库，不扩展到复杂协作能力。这样既能补齐作者站内写作链路，又能控制范围，不破坏当前 file-driven 架构。
