## 二审结论（reviewer-gemini）

### 补充项

- [必改] 发布态与发布按钮存在“双重切换”歧义，容易把用户刚切好的状态又反向提交。`PublishToggle` 会先直接改本地 `published` 状态，而右侧主按钮 `发布 / 取消发布` 在提交时又执行一次 `!form.published`。结果是：用户若先点卡片上的“已发布/草稿”切换，再点主按钮，最终提交的状态会反转，和界面当前所见不一致。这会直接影响发布/取消发布路径可理解性，也会制造误操作风险。见 `src/components/writer/workbench-form.tsx:79`、`src/components/writer/workbench-form.tsx:208`、`src/components/writer/publish-toggle.tsx:18`。
- [必改] 已发布文章的“预览”路径会落到 404，保存并预览在发布后同样失效。编辑页始终展示“预览当前已保存版本”链接，保存并预览按钮也始终可点，但预览页明确对 `post.published` 执行 `notFound()`。这导致文章一旦发布，编辑页上的预览入口就变成坏链路；作者无法理解“预览”和“公开查看”的关系。见 `src/app/dashboard/write/[slug]/page.tsx:25`、`src/components/writer/workbench-form.tsx:137`、`src/app/dashboard/preview/[slug]/page.tsx:24`。
- [可改] 当前错误提示全部落在顶部全局告警，未贴近具体字段。虽然 API 已返回清晰中文报错，但像 `title 不能为空`、`excerpt 不能为空`、`slug 只能包含...` 这类表单错误仍需用户自己回看字段定位，编辑体验偏生硬，不符合方案里“字段错误贴近字段显示”的目标。见 `src/components/writer/workbench-form.tsx:176`、`src/lib/author-posts.ts:137`。
- [可改] `/dashboard/write` 列表只展示有效文章，待修复内容不会在该页出现，作者需要回到 `/dashboard` 才能发现 invalid 文件。对于“在线写作工作台”这个命名来说，入口分散，恢复路径不够顺手。见 `src/app/dashboard/write/page.tsx:7`。

## 三审结论（reviewer-minimax）

### 安全风险项

- [✔] **公开读取路径隔离完整**：`src/lib/posts.ts` 中 `getPostBySlug()` 调用 `findPostBySlug()` 时默认不传 `includeDrafts`，只返回 `published=true` 的文章；`src/app/blog/[slug]/page.tsx` 用此函数，永远不可能命中草稿。公开路由与作者态路由彻底分层。
- [✔] **作者侧读取路径隔离完整**：`src/app/dashboard/preview/[slug]/page.tsx` 使用 `getPostBySlugForAuthor(slug)`（传 `includeDrafts: true`），可预览草稿和已发布文章。路由本身有 `requireAuthorPageSession()` 守卫，非登录/非白名单用户无法访问。
- [✔] **API 权限守卫完整**：POST / GET / PUT 三个入口统一调用 `requireAuthorApiSession()`，未登录返回 401 `{ error: "请先登录" }`，非白名单返回 403 `{ error: "你没有写作权限" }`，不泄露内部信息。
- [✔] **文件系统边界封闭**：写入路径仅限 `POSTS_DIR`（即 `content/posts/`），API body 不接受任意路径参数；`buildPostFilePath()` 硬编码拼接，无注入面。
- [✔] **Slug 冲突检测覆盖双文件同 slug 场景**：`ensureSlugNotUsedByOtherFile()` 在 create 和 update 两条链路上均被调用，排除当前文件自身后遍历所有已解析的 post 条目，匹配 slug 即抛 409；可防止"不同日期文件名共享同一 slug"导致的公开路由歧义。
- [✔] **Mtime 冲突检测有效**：`updateDraft` 读取文件当前 `mtimeMs`，与前端传来的 `lastKnownMtimeMs` 做差值比较（容差 1ms），差异即抛 409，不静默覆盖。原子写入通过临时文件 + rename 实现，写入失败不破坏原文件。
- [✔] **发布动作不旁路校验**：`submit("publish-toggle")` 走统一 PUT 链路，发布状态切换复用同一 `validateInput` → `updateDraft` 路径，不会绕过 slug 冲突检测或 mtime 检测。
- [~] **文件系统权限依赖**：所有写操作依赖运行进程对 `content/posts/` 有写权限，reviewer 在验收时需确认部署环境具备此条件（本审阅范围内无法验证）。

### 中文表达项

- [✔] **字段错误提示准确**：所有 400 校验错误均为中文，`title 不能为空`、`date 格式须为 yyyy-MM-dd`、`slug 不能为空`、`slug 只能包含小写字母、数字和中横线`、`excerpt 不能为空`，与用户输入字段直接对应。
- [✔] **字段级错误映射已实现**：`FrontmatterFields` 组件接收 `FieldErrors` 类型 `{ title?, date?, slug?, excerpt? }`，`workbench-form.tsx` 的 `mapFieldError()` 将错误消息按关键词匹配分发到对应字段下显示，title/date/slug/excerpt 四个高频字段均有贴字段错误展示。
- [✔] **发布状态展示清晰**：`PublishToggle` 组件只读展示，不执行切换动作，消除双重反转歧义；草稿显示"当前为草稿，只在作者侧可见"，已发布显示"当前为已发布，公开博客会读取它"。
- [✔] **编辑页入口文案准确**：已发布文章展示"查看公开页 →"，草稿展示"预览当前已保存版本 →"，区分两种查看语义；新建页禁用发布按钮并展示"新建后可发布"。
- [✔] **预览页提示语境适配**：预览页根据 `post.published` 状态分别展示"这是作者侧预览；文章已发布，公开页也会读取当前保存版本"和"这是草稿预览，内容尚未公开发布"，帮助作者区分当前预览语境。
- [✔] **冲突提示语义准确**：409 错误有"文件名冲突，请修改 slug 或日期"（目标路径已存在）、"slug 已被其他文件占用，请修改 slug"（slug 被占用）、"文件已更新，请刷新后重试"（mtime 冲突），三种场景各有明确提示。
- [~] **非阻塞建议**：`workbench-form.tsx` 中"保存并预览"按钮在已发布文章时仍然可点（跳转 preview 页），逻辑上可工作，但用户可能会困惑为何不是"查看公开页"。建议保持现状（二审已要求改为"查看公开页"并在编辑页体现），当前按钮文案在已发布态下略有歧义，但不影响安全或数据完整性。
