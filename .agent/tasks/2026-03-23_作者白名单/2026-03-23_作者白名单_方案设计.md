# 作者白名单 方案设计

> 日期：2026-03-23
> 任务：作者白名单
> 负责人：planner-codex

## 目标

限制只有白名单内的 GitHub 用户才能登录进入 dashboard。未在白名单内的 GitHub 用户即使完成 OAuth 登录，也会看到拒绝页面。

## 背景

当前 dashboard 只有一层保护：必须登录。但任何有 GitHub 账号的人都能登录进来，这不够安全。需要加一层白名单，只允许作者本人进入后台。

## 当前状态

- `auth.ts`：JWT session，`token.sub` 已映射为 `session.user.id`
- `dashboard/page.tsx`：只检查 `session?.user` 是否存在，存在就放行
- 白名单：**不存在**

## 技术方案

### 方案 A：在 auth callback 中拦截（推荐）

在 NextAuth `callbacks.jwt` 或 `callbacks.session` 里做白名单判断。

- 读取环境变量 `ALLOWED_GITHUB_USERS`（逗号分隔的 GitHub 用户 ID 列表）
- 在 `jwt` callback 里验证 `token.sub` 是否在白名单内
- 如果不在，直接返回空 session 或抛出错误
- 优点：拦截发生在服务端，安全性高，逻辑集中
- 缺点：需要重启服务才能更新白名单

### 方案 B：在 dashboard 页面判断

在 `dashboard/page.tsx` 里读取环境变量并判断。

- 优点：更新 env 后 dashboard 重新渲染即可感知
- 缺点：逻辑分散在页面层，需要重复判断，安全性较低

### 方案 C：数据库存储白名单

引入数据库，动态管理白名单。

- 优点：可动态增删用户，无需重启
- 缺点：V1 阶段不引入数据库，不符合"V1 文件驱动"原则

**结论：采用方案 A**（安全性高，逻辑集中，V1 最合适）

## 白名单管理方式

环境变量：`ALLOWED_GITHUB_USERS`

格式：逗号分隔的 GitHub 用户 ID

```
ALLOWED_GITHUB_USERS=123456,789012
```

用户的 GitHub ID 可从当前 dashboard 页面的"用户 ID"字段看到（就是 `session.user.id`）。

V1 阶段通过直接修改 `.env.local` + 重启服务来管理白名单。后续 V2 再考虑动态管理界面。

## 范围

### 做
- 在 `auth.ts` 的 `callbacks.jwt` 中加入白名单验证
- 在 `auth.ts` 中加入 `isWhitelisted` 辅助函数
- 环境变量 `ALLOWED_GITHUB_USERS` 配置说明
- 未授权用户看到友好的拒绝页面（而非报错）
- dashboard 页面显示当前用户是否在白名单内

### 不做
- 动态白名单管理界面
- 数据库存储白名单
- 邮件/通知等复杂权限体系

## 风险与取舍

### 风险 1：白名单为空
如果 `ALLOWED_GITHUB_USERS` 为空或未配置，所有用户都会被拒绝。
**应对**：默认允许配置为空时报警，或在文档中明确说明必须配置。

### 风险 2：不知道自己的 GitHub ID
用户可能不知道自己的 GitHub ID。
**应对**：在 dashboard 登录后显示 GitHub ID，用户可用此 ID 配置白名单。

## 验收标准

1. 白名单内用户：正常登录，正常进入 dashboard
2. 白名单外用户：登录成功，但访问 dashboard 时跳转到拒绝页面
3. `ALLOWED_GITHUB_USERS` 未配置时：行为与当前一致（不拦截，作为开发容错）
4. `npm run lint` 和 `npm run build` 均通过
5. 登录/退出流程完整，不破坏现有功能

## 结论

采用方案 A，在 NextAuth jwt callback 中集中判断，env 管理，V1 最简路径。
